# Tests to Drop

## Rules for dropping

- **Trivial formatting**: test is just "multiply and format"; a UI glance catches regressions faster than the test suite
- **No real logic**: the code path has no branching, filtering, or domain math — it can't silently break in a non-obvious way
- **Strict subset**: another test already exercises the same branches and more — keeping both adds maintenance cost with no extra coverage
- **Strict subset across layers**: if a higher-level test (component, integration, or engine) already verifies a behavior AND the wiring that produces it, a lower-level test checking only the behavior is redundant — drop the lower-level test. Applies across files: e.g. an engine test covering multi-effect dispatch makes a data test re-asserting the same pattern redundant
- **TypeScript-enforced shape**: tests that assert structural properties (field presence, enum membership, uniqueness) already guaranteed by `satisfies` or explicit type annotations — the compiler is the right layer, drop the test
- **Value duplication across layers**: when a higher-layer test (hook, component, integration) re-asserts concrete values that a lower-layer unit test already pins, trim it to verify only the _wiring_ — that the value flowed through — and delegate the value table to the unit test. Don't maintain the same expected values in two places. This is a trim, not a delete: the wiring assertion still earns its keep (e.g. a `useTeam` test asserting `loadouts[0]` equals `loadoutFromTemplate(...)` rather than re-listing each resolved id that `template.test.ts` already checks)
- **Pure-function determinism restatement**: a test that asserts a pure function returns the same output for the same input (`f(x) === f(x)`) is testing referential transparency, which is structural — same spirit as the TS-enforced-shape rule. Drop it; keep the tests that pin _what_ the output is (e.g. an encode/decode roundtrip earns its keep, but a standalone `"encode is deterministic"` does not)
- **Lookup-contract null-guard pairs**: a named sub-case of "no real logic". When the function under test is a one-line registry lookup (`arr.find(x => x.id === id) ?? null`), the "returns the match for a known id" + "returns null for an unknown id" pair tests the language's `find`/`??`, not domain logic — it can't silently break in a non-obvious way. Drop both. Keep lookups that filter, match across a relation, or transform (e.g. `listWeaponsByType`, `getEchoSetForEcho`)
- **Cosmetic not-found / empty fallbacks**: tests pinning a missing-data placeholder (`"—"`, `"?"`, `#888`, null bundle) don't earn their keep — a silent blank is harmless and obvious, and only a crash would matter, which these don't guard. Drop the not-found case, keep the happy path. Exception: a fallback that changes a real number (null-vs-0 damage) is real logic — keep it
- **Re-render isolation / render-count assertions**: tests that mount a probe, count renders, and assert an unrelated change didn't re-render it (e.g. a `selectAtom`/`memo` bailout) don't earn their keep — they pin a performance characteristic, not a behavior, and break on incidental render-timing changes while a real correctness regression slips past. Verify re-render scope by **audit** (reason about the atom/selector dependencies and equality, or profile), not by a committed test. Keep tests that assert _what_ a slot/derived atom yields, not _how many times_ it renders
- **Glance-visible failure**: the master question for component tests — if this broke, would the page visibly break on first use, or degrade silently / show wrong data? Only silent or wrong-data failures earn a test. Dead buttons, missing labels, misaligned tables, and empty lists announce themselves: drop their tests
- **Static style pins**: assertions on exact colors/rgba channels, shared class skeletons, or "theme A ≠ theme B" inequalities pin restyle-brittle facts a UI glance catches. Keep a style assertion only when the style is the sole observable output of a domain branch (threshold reached → pill fill). Feedback styling (success flash, lock state, stale dimming) is cosmetic even when branch-driven — drop
- **Tombstones**: a test or assertion whose point is that removed code stayed removed — `not.toHaveProperty`, "no X button remains", spying that a replaced API isn't called, negative widget inventories ("no BASIC chip"). Proof that a refactor landed dies with the PR that landed it. When the negative half rides inside an otherwise-useful test, trim the assertion and keep the test
- **Pass-through wiring**: click fires the obvious callback, children render, a prop reaches the DOM verbatim — JSX can't miswire these non-obviously, and a break is glance-visible. Drop, including verbatim prop forwarding like `disabled={flag}`. One-line glue closures (a return value piped into an atom) are left to review, or extracted into a pure function and unit-tested when the component is next touched — never mount-tested
- **Cosmetic interaction**: hover/tooltip/focus/toggle-state niceties and acknowledgment state machines (flash-and-revert buttons, frozen hover readouts) fail self-evidently on first use — drop. Keep an interaction test only when the interaction commits data (`rename(id, name)`) or selects among competing actions (row click selects, must not launch)
- **Mapping-table enumeration**: for a flat key → value table, keep one representative row plus every hazard row: the default/fallback branch, a row that changes the code path (name resolution), combination rules (suffixes), and aliasing collisions (`echo-set.` must not match as `echo.`). Drop the remaining rows, and drop parallel enumerations of the same predicate over sibling fields or resources (three stats × one formula → test one)
- **Empty/default-state guards**: guard clauses that return the default or empty output (empty input → empty output, no data → section omitted, all-zero → null badge) are bloat — drop all of them, not just one of an enumerated set. The null-vs-0 exception above still holds: a fallback that writes a real number into real data keeps its test
- **Vacuous fixture**: the fixture cannot exhibit the property in the title (an "order preserved" test whose fixture yields one item) — the test passes forever while verifying nothing. Delete it; rebuild the fixture only if the property is critical
- **Presentation ordering**: tests pinning display order (intro-first sorting, emission order within a group) are low-priority polish — drop unless the order is domain-critical
- **Layout-structural counts**: cell counts against a shared column constant, divider presence, alignment when an optional column toggles — misalignment is the most glance-visible failure there is. Drop
- **Which-field guards (the keep archetype)**: a test pinning the choice between two plausible, diverging data sources (skillCategory vs skillType, grouping vs category, parent field vs stage field) guards silent wrong data — the strongest kind of keep, worth a fixture built specifically so the axes diverge
- **Idempotent spurious-fire negatives**: when a gate guards an action that is harmless if fired spuriously (re-running a deterministic sim, re-invalidating an already-stale cache), the "does not fire when flag X is false" rows are efficiency guards, not correctness guards — drop them, keeping at most one representative negative per flag on its primary path. The positive rows (missed fire → silently stale output) keep their tests
