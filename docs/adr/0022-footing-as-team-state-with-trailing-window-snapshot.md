# Footing as team state with per-character carried footing on the event stream

Stages gain an optional `footing: "ground" | "air" | { launch: number } | { land: number }` field declaring the team's vertical-position state across the stage. Sustained values are terse strings; transitions are objects carrying the in-stage **commit frame** — the action-frame within the stage at which the vertical-position change actually occurs in the animation (e.g. the frame Jiyan's body leaves the ground in his launch animation, not the frame the swing input is accepted). The commit frame is a point-of-no-return: a swap-cancel whose `actionTime` lands before it does **not** complete the transition, so the launch (or land) never happens.

`Jump` is authored as `{ launch: <action-frame> }` (ground→air); `Dodge` omits `footing` entirely and is footing-transparent. Different-footing variants of the same conceptual move (e.g. a stage that exists in both a ground and an aerial form) are authored as separate fundamental Stages rather than a polymorphic stage with footing parameters — they almost always carry different Damage Entries anyway.

**Deployment State.** A character is in exactly one of: **On-field** (deployed/controlled), **In-trailing** (swapped out, its last stage's Trailing Window still open — duration = the stage's `actionTime`), or **In-reserve** (window passed; inert). Footing is _live_ while On-field or In-trailing; an In-reserve character is `ground`.

**Two footing values on `FootingTracker`** (a collaborator sibling to `OnFieldTracker`, same mutation-versioned shape, entirely orthogonal — swap inference and footing inference do not read each other):

1. **Team footing** — the field's vertical state = the On-field character's footing. A _fresh_ swap-in inherits it (swapping while airborne keeps the incoming character airborne). On-field launch/land commits set it via `applyStageFooting`.
2. **Carried footing** — `Map<characterId, "ground" | "air">`, a per-character override. Set by a launch/land commit firing while the owner is In-trailing (off-field), and by the window-end reset. Consumed at the owner's next swap-in: if present it is used **and promoted to team footing**; otherwise team footing is inherited.

**Footing commits are stream events (instant, no easing).** A swap stage's launch/land enqueues a footing commit at its commit frame onto the one frame-ordered event stream ([ADR-0028](0028-emithits-as-frame-honest-worklist.md)); on resolve it sets the owner's **carried** footing (the owner is off-field, so team footing is untouched). The carry fires for **every** swap launch/land, not only those whose commit frame lands past the variant advance — the commit is owner bookkeeping (what the owner resumes on a swap-back), while the few frames around the advance only decide what a _fresh_ swap-in inherits, which is the on-field `applyStageFooting` flip. Commits obey the same drop/pad rules as trailing hits: a same-character cancel-capable re-entry before the commit frame drops it (the launch never happened — the point-of-no-return made explicit); a non-cancel re-entry pads past it so it fires.

**In-trailing → In-reserve reset.** An airborne In-trailing character returns to `ground` when its Trailing Window passes without a swap-back — a **window-end reset** stream event at `stageStart + actionTime` setting carried `ground`. A re-entry _within_ the window cancels it (carried `air` survives → the swap-back enters airborne and pays fall); a re-entry _at or after_ window-end lets it fire first (the character is benched to `ground` before it takes the field again). Carried footing is read only at the owner's own swap-in, and every entry drains the stream to its arrival frame before reading, so the lazy reset is never observed stale.

**Validity (timeline-authoring-only).** The four mismatch cases against current footing on stage entry:

| current footing | stage requires                            | result                                  |
| --------------- | ----------------------------------------- | --------------------------------------- |
| `air`           | `ground` (or implied entry of `{ land }`) | soft — `fall` padding fires (see below) |
| `ground`        | `air` (or implied entry of `{ launch }`)  | hard error — "launch/jump required"     |
| `air`           | `{ launch }`                              | hard error — "already airborne"         |
| `ground`        | `{ land }`                                | hard error — "nothing to land from"     |

The asymmetry is physical: gravity is free (falls auto-pad), launches cost an animation (must be explicit). Validation lives in `validateTimeline` only; the engine trusts valid timelines. This parallels how the Prior-Stage Gate is already validator-only. `validateTimeline`'s exit-footing inference is variant-aware for `{ launch }` / `{ land }` stages: under `cancel` / `instantCancel`, exit equals the transition target only when the authored commit frame is `≤ resolved variant advance` (the cursor reached the point-of-no-return before the cancel); otherwise exit equals entry footing. Under `swap`, the cursor's exit always equals entry footing — the transition, if any, fires later via the stream commit.

**Fall padding.** A global player-level setting `fallFrames` (default 21 ≈ 0.35s @ 60fps) joins `reactionDelay`, `swapFrames`, and `variantFloor` in `wuwa.settings`. It surfaces as the `fall` component of Padding Delay — additive (does not compete with the `react`/`floor` max), not subject to `variantFloor` (input-duration minimums are orthogonal to gravity), and **attaches to the current entry's startup delay**, not the previous entry's `advance` — a deliberate divergence from `pad`'s "extend previous entry" convention. The reasoning: `pad` exists because A's trailing hits are still running so A's animation legitimately takes longer; `fall` exists because B's character is mid-air when B wants to start, so the delay is B's startup. The current-entry attribution keeps the UI legible in cross-character cases (Verina's ground stage following an aerial Jiyan swap pays the `fall` — showing it on Verina's row matches her landing animation, where showing it on Jiyan's swap stage would not).

## Considered Options

- **Per-Damage-Entry footing tag, no Stage-level field, no tracker.** Rejected. Validity is a stage-to-stage transition concept (the next stage's legality depends on the previous stage's state), not a per-hit one. Modeling it at the hit level forces every consumer (validator, engine, padding calculator) to scan damage arrays and reconstruct stage state, and the fall-padding question becomes ambiguous (which hit triggers fall?). Stage-level matches the granularity of validity. Movement Stages (`Jump`) also have no damage entries — they'd need a phantom entry just to hang the tag.
- **Per-Skill-Type footing implied by a closed mapping (e.g., aerial-Skill-Type values).** Rejected. Skill Type is already the engine's "what kind of skill produced this" axis (amps, skill-type bonus, hitLanded filters); overloading it with vertical-position semantics conflates two orthogonal dimensions, and characters with both grounded and aerial Heavy Attacks would need new Skill Types per footing — an explosion the closed taxonomy is designed to prevent.
- **Separate `startsIn` and `endsIn` fields on Stage.** Rejected as redundant authoring surface. The cross-product is only four valid cells (gg, aa, ga, ag), and the two-field form invites nonsense combos at the type level that the closed union prevents by construction.
- **Add a fifth value `"any"` or `"preserve"` to denote footing-transparent stages.** Rejected. Every consumer would have to special-case the value (validator, tracker, padding calculator, UI display). Omitting the field entirely — already the convention for optional Stage fields like `variants` — communicates the same intent with no extra code paths. A doc note in CONTEXT.md clarifies that omission is intentional, not missing-data.
- **Default omission to `"ground"` (instead of footing-transparent).** Rejected. Dodge's behavior is genuinely footing-transparent — a single button-press in-game that preserves whichever state you were in. Defaulting omission to `ground` would also force a silent migration of every non-aerial-aware stage to explicit `"ground"`, and any missed retrofit becomes a soft-fall padding hazard the moment an aerial stage precedes it. Transparent omission preserves the existing meaning.
- **Two distinct `Dodge` Movement Stages (`Dodge (ground)` / `Dodge (air)`), sidebar groups them under one label.** Considered seriously. Rejected for the same Dodge-is-transparent reason — forcing two stages would require the sidebar to resolve which to inject based on current footing (couples UI to tracker state), or the user to manually pick (UX worse than transparent passthrough).
- **Per-character footing throughout (no team-level state).** Rejected. Scenario: charA plays a launch stage and goes aerial, swaps out, charB lands, charB swaps to charC, charC plays grounded rotations for 10 seconds, charA re-enters via a normal cast. Under naïve per-character state, `charA.footing` is still `air` (last write) — charA re-enters aerial after 10 seconds off-field. In-game wrong. Team footing carries the "what the player has actually been seeing on the active character" semantic, which is what a re-entering In-reserve character physically should read. The carried map is an override scoped to the In-trailing window, not a permanent per-character value — the window-end reset is exactly what prevents the stale-air bug.
- **Per-character trailing-window footing _snapshot_ dict on `FootingTracker` (the original design — shipped, then removed).** A `{characterId → snapshot}` dict populated when a commit fired while the owner was off-field, consumed and promoted to team at swap-in. Removed once built, because the off-field producer was never actually wired: the only producer fired at the owner's **re-entry** and the only consumer ran in the **same loop iteration**, so the dict's entries never spanned a tick — it was a redundant second per-character copy of a fact the trailing window's own pending-commit already held. The design conflated "fire the commit off-field" (which would need the snapshot) with "fire the deferred commit at re-entry" (the value is already in hand) — and only the latter was built.
- **Team-global footing only, no per-character state at all (the interim state after the snapshot removal — held, then reversed).** Correct while footing was read **only for the on-field character**: an off-field character's vertical divergence could stay parked in its pending commit and apply at re-entry, when "their footing" _is_ team footing. Reversed when the deferred consumer arrived — characters that **act while off-field** (the In-trailing state on the one frame-ordered event stream) made commits genuinely fire off-field. Carried footing re-introduces per-character state, but as the live _result_ of stream commits rather than the original parallel snapshot dict; the window-end reset (impossible in the snapshot design) is what keeps it from regressing to the stale-air bug.
- **Eager commit at stage resolution (4-string encoding: `"ground" | "air" | "launch" | "land"`).** Rejected. Collapses two distinct frames — the player's commit point and the animation's actual transition point — into one. In-game, a sufficiently early swap-cancel of a launch stage does **not** reach the point-of-no-return frame, so the launch never happens. The 4-string model has no way to express this; the frame-bearing transition object makes the point-of-no-return explicit in authoring.
- **Two fields on Stage (`footing` + `footingFrame: number`).** Rejected. The frame is meaningful **only** for transitions; encoding it as a separate field invites authoring sustained `"ground"`/`"air"` with a stray `footingFrame: 0` and forces every reader to special-case "is this field meaningful right now?" The discriminated union puts the frame where it has meaning and nowhere else.
- **Always-object form (`{ state: … } | { transition: …; at: number }`).** Rejected. Uniform shape, but `{ state: "ground" }` is ceremony around what used to be `"ground"`. Every existing stage gets noisier for no behavioral gain.
- **String union with optional `at` defaulting to "end of advance".** Rejected. Most migration-friendly, but hides the load-bearing decision behind a default — the `at?: number` optional re-introduces the eager-commit bug, just with extra steps.
- **Eager commit at swap time when the launch frame is "close enough" to `swap.actionTime`.** Rejected. The whole motivation for the mid-stage commit frame is modeling the point-of-no-return correctly; an eager-if-close-enough heuristic re-introduces the eager-commit bug under a fuzzy threshold. Authors who want a swap-cancel to feel like a launch-swap can author `swap.actionTime ≥ footing.launch`.
- **Make `FootingTracker` aware of swap events (consumes `swapIn`/`swapOut` from `OnFieldTracker`).** Rejected. Coupling the two trackers creates an event-ordering dependency the existing engine doesn't have. The on-field check at footing-event dispatch reads `OnFieldTracker` at call-site (engine-layer), but the two trackers themselves remain siblings with no cross-references.
- **Engine-runtime assertion of footing validity (defense-in-depth alongside `validateTimeline`).** Rejected. No other Stage-level validity rule is engine-asserted; the precedent is that the engine trusts the validator. Doubling the rule surface buys no end-user value and creates two places to keep in sync.
- **Air→ground as a hard error (require explicit `land` Movement Stage to bridge).** Rejected. The game itself auto-recovers from aerial state via gravity — there's no "Land button" the player presses; they just fall. Modeling this as a hard error would force every aerial-to-grounded sequence to author a `land` stage that doesn't correspond to any real player input, polluting timelines for no analytic gain. Soft-with-padding matches the actual mechanic.
- **Ground→air as a soft warning with auto-inserted launch padding.** Rejected for the symmetric reason — there is no gravity equivalent that lifts the character off the ground without input. Going aerial requires an explicit `Jump` or character-specific launch move; silently padding the transition would invent frames the player never spent.
- **Attach `fall` padding to the previous entry's `advance` (symmetric with `pad`).** Rejected. The cross-character case is load-bearing: charA aerial swap-variant → charB grounded stage. `pad` on the previous entry would put a `+0.35s fall` on charA's row, but charA isn't the one falling — charB is. The current-entry attribution matches the actual physical event the padding models.
- **Fall padding subject to `variantFloor`.** Rejected. `variantFloor` models the physical minimum input duration a player can execute — a finger-on-button constraint. Fall padding models gravity — a world-state constraint. They are orthogonal physical phenomena and the floor's max-semantics would let one mask the other, producing under-padded sequences whenever both apply. Both should accumulate.
- **`fallFrames` per-character (heavier characters fall slower).** Rejected as overmodeling. The sim doesn't track height, mass, or any other variable that would meaningfully differentiate character fall speeds, and the in-game variance is small enough that a single global default captures the phenomenon well enough to drive optimization decisions. If a future character genuinely needs a different fall constant, the field can be added on Character as an override without changing the global default.

## Consequences

- `Stage` gains an optional `footing?: "ground" | "air" | { launch: number } | { land: number }` field. Existing stages without `footing` are treated as transparent — no migration is needed for non-aerial content.
- `src/data/movement.ts`: `Jump` is `{ launch: <action-frame> }`; `Dodge` omits `footing`. No other Movement Stage fields change.
- `Settings` carries `fallFrames` (default 21) alongside `reactionDelay`, `swapFrames`, `variantFloor`; the `useLocalStorage` merge path handles older saved settings via the default, and a settings-UI control surfaces it.
- `FootingTracker` (`src/lib/engine/footing-tracker.ts`): `teamFooting()`, `setTeam(f)`, `carriedFor(id)`, `setCarried(id, f)`, `takeCarried(id)`, `mutationVersion()`.
- `FootingModule`: `team()`, `applyStageFooting(footing, stageDuration)` (on-field commits), `commitFor(id, exitFooting)` (a stream commit/reset sets the owner's carried footing), `resolveEntry(id)` (the swap-in read: carried override if any — promoted to team and consumed — else inherited team footing). Dependency direction is one-way and acyclic: the swap-stage scheduling calls into Footing at fire time; Footing knows nothing about swaps or trailing tails.
- The simulation enqueues each swap stage's footing commit and window-end reset onto the pending stream (built by `partitionStage`'s `footingChanges` in `trailing-window.ts`); the `Schedule`'s arrival handling invalidates the commit on a cancel-capable collision, cancels the reset on a within-window re-entry, and pads on a non-cancel collision; the drain applies both via `commitFor`.
- `validateTimeline` gains a footing-validity pass: a current-footing cursor over the entry sequence, row errors for the hard cases, fall padding surfaced as a row annotation (not an error), exit-footing inference variant-aware as described above.
- **Swap-back-pays-fall:** a re-entry _within_ the launcher's window enters airborne and pays `fall`; a swap-back _after_ `actionTime` benches to `ground` first (no fall) — the character has "landed in the background."
- `Padding Delay`'s tooltip carries a `fall` component, additive alongside `pad` (the `react` vs. `floor` mutex is unchanged).
- New domain terms under Core simulation: **Footing**, **FootingTracker**, **Fall Frames**, **Deployment State** (On-field / In-trailing / In-reserve), **Carried Footing**.
- Footing transitions do not fire `tookOff` / `landed` engine events. No currently authored buff triggers on the transition itself (buffs that care about air do so via skill-type / stage-id filters on the already-tagged air stage). A future ADR can introduce events when the first buff needs them. Likewise, In-trailing hits remain footing-blind (ADR-0028's invariant): no emission resolves a footing-conditional buff.
