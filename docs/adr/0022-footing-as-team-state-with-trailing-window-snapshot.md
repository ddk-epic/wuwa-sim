# Footing as team-level state with per-character trailing-window snapshot override

Stages gain an optional `footing: "ground" | "air" | { launch: number } | { land: number }` field declaring the team's vertical-position state across the stage. Sustained values are terse strings; transitions are objects carrying the in-stage **commit frame** — the action-frame within the stage at which the vertical-position change actually occurs in the animation (e.g. the frame Jiyan's body leaves the ground in his launch animation, not the frame the swing input is accepted). The commit frame is a point-of-no-return: a swap-cancel whose `actionTime` lands before it does **not** complete the transition, so the launch (or land) never happens.

The engine tracks current team footing as a single value, updated by `launch` / `land` commit events fired from the [Trailing Window](0018-swap-variant-as-trailing-window.md) when the owning character is on-field at the commit frame. A `FootingTracker` collaborator owns this state — sibling to `OnFieldTracker`, same mutation-versioned shape, but entirely orthogonal: swap inference and footing inference do not read each other. `Jump` is authored as `{ launch: <action-frame> }` (ground→air); `Dodge` omits `footing` entirely and is footing-transparent. Different-footing variants of the same conceptual move (e.g. a stage that exists in both a ground and an aerial form) are authored as separate fundamental Stages rather than a polymorphic stage with footing parameters — they almost always carry different Damage Entries anyway.

**Validity (timeline-authoring-only).** The four mismatch cases against current team footing on stage entry:

| current team footing | stage requires                            | result                                  |
| -------------------- | ----------------------------------------- | --------------------------------------- |
| `air`                | `ground` (or implied entry of `{ land }`) | soft — `fall` padding fires (see below) |
| `ground`             | `air` (or implied entry of `{ launch }`)  | hard error — "launch/jump required"     |
| `air`                | `{ launch }`                              | hard error — "already airborne"         |
| `ground`             | `{ land }`                                | hard error — "nothing to land from"     |

The asymmetry is physical: gravity is free (falls auto-pad), launches cost an animation (must be explicit). Validation lives in `validateTimeline` only; the engine trusts valid timelines. This parallels how `requiresStageId` and `comboAllows` are already validator-only.

`validateTimeline`'s exit-footing inference is variant-aware for `{ launch }` / `{ land }` stages: under `cancel` / `instantCancel`, exit equals the transition target only when the authored commit frame is `≤ resolved variant advance` (the cursor reached the point-of-no-return before the cancel); otherwise exit equals entry footing. Under `swap`, the cursor's exit always equals entry footing — the transition, if any, fires later via the trailing window.

**Fall padding.** A new global player-level setting `fallFrames` (default 21 ≈ 0.35s @ 60fps) joins `reactionDelay`, `swapFrames`, and `variantFloor` in `wuwa.settings`. It surfaces as the `fall` component of Padding Delay — additive (does not compete with the `react`/`floor` max), not subject to `variantFloor` (input-duration minimums are orthogonal to gravity), and **attaches to the current entry's startup delay**, not the previous entry's `advance` — a deliberate divergence from `pad`'s "extend previous entry" convention. The reasoning: `pad` exists because A's trailing hits are still running so A's animation legitimately takes longer; `fall` exists because B's character is mid-air when B wants to start, so the delay is B's startup. The current-entry attribution keeps the UI legible in cross-character cases (Verina's ground stage following an aerial Jiyan swap pays the `fall` — showing it on Verina's row matches her landing animation, where showing it on Jiyan's swap stage would not).

**Team-global with per-character trailing-window override.** Two pieces of state live on `FootingTracker`:

1. **Team footing** — single value (`ground | air`), the on-field character's current vertical state. Updated by `launch` / `land` commit events fired from the trailing window when the owning character is on-field at the commit frame.
2. **Per-character trailing-window footing snapshot** — `{characterId → snapshot}` dict, populated when a `launch` / `land` commit fires while the owning character is _off-field_ (mid-trailing-window after a swap-cancel that crossed the point-of-no-return). Cleared when the trailing window expires or the character re-enters.

On swap-in, the engine consumes the incoming character's snapshot if any and **promotes it to team footing** (the on-field invariant: team footing reflects whoever is currently on-field) before the entry's footing validity check. If no snapshot exists, the incoming character simply inherits current team footing. After re-entry, subsequent `launch` / `land` commits update team footing in place via the on-field path.

This composition resolves a real tension: footing has to be team-global so that "aerial swap to a different character lands them aerial too" composes naturally — when Jiyan reaches his launch commit frame and team flips to `air`, any swap to Verina after that inherits `air`. But it also has to remember that a swapped-out character whose own commit fires _after_ they left the field is still in their own aerial state, so when **that same character** re-enters during their trailing window, the snapshot promotes back to team. The snapshot is the bookkeeping minimum that captures this without making the entire system per-character.

**Trailing Window event scheduling.** When a stage with `{ launch }` / `{ land }` is scheduled, the Trailing Window populates a `pendingFooting: { atFrame, exitFooting }` alongside its trailing hits — at most one per in-flight stage by construction (one launch/land frame per stage). The state shape becomes `TrailingEntry { hits, pendingFooting? }`. Cancel-capable drop semantics from [ADR-0018](0018-swap-variant-as-trailing-window.md) apply identically to the footing event: a same-character cancel-capable re-entry that drops events past `newEntry.startFrame` also drops `pendingFooting` if `pendingFooting.atFrame ≥ newEntry.startFrame` (no team flip, no snapshot — the launch never happened). The non-cancel-capable pad-extension target frame becomes `max(lastHitFrame, pendingFooting?.atFrame ?? -∞)`.

**Worked example.** charA does a launch swap-variant stage with `swap.actionTime ≥ footing.launch` (the cancel fires after the commit frame). At the launch commit frame, charA is already off-field — `snapshotFor(charA, "air")`. Team footing remains `ground`. charB swaps in inheriting team `ground`. charB plays grounded stages. charA's swap-back occurs _within_ their own trailing window: `consumeSnapshot(charA)` returns `air`, the on-field invariant promotes it, team is now `air`. If instead `swap.actionTime < footing.launch`, the cancel fires before the commit frame and the entire `pendingFooting` drops — charA's swap-back inherits whatever team footing has become (likely `ground`), correctly modeling the in-game "never actually launched" case.

## Considered Options

- **Per-Damage-Entry footing tag, no Stage-level field, no tracker.** Rejected. Validity is a stage-to-stage transition concept (the next stage's legality depends on the previous stage's state), not a per-hit one. Modeling it at the hit level forces every consumer (validator, engine, padding calculator) to scan damage arrays and reconstruct stage state, and the fall-padding question becomes ambiguous (which hit triggers fall?). Stage-level matches the granularity of validity. Movement Stages (`Jump`) also have no damage entries — they'd need a phantom entry just to hang the tag.
- **Per-Skill-Type footing implied by a closed mapping (e.g., aerial-Skill-Type values).** Rejected. Skill Type is already the engine's "what kind of skill produced this" axis (deepens, skill-type bonus, hitLanded filters); overloading it with vertical-position semantics conflates two orthogonal dimensions, and characters with both grounded and aerial Heavy Attacks would need new Skill Types per footing — an explosion the closed taxonomy is designed to prevent.
- **Separate `startsIn` and `endsIn` fields on Stage.** Rejected as redundant authoring surface. The cross-product is only four valid cells (gg, aa, ga, ag), and the two-field form invites nonsense combos at the type level that the closed union prevents by construction.
- **Add a fifth value `"any"` or `"preserve"` to denote footing-transparent stages.** Rejected. Every consumer would have to special-case the value (validator, tracker, padding calculator, UI display). Omitting the field entirely — already the convention for optional Stage fields like `variants` and `comboAllows` — communicates the same intent with no extra code paths. A doc note in CONTEXT.md clarifies that omission is intentional, not missing-data.
- **Default omission to `"ground"` (instead of footing-transparent).** Rejected. Dodge's behavior is genuinely footing-transparent — a single button-press in-game that preserves whichever state you were in. Defaulting omission to `ground` would also force a silent migration of every non-aerial-aware stage to explicit `"ground"`, and any missed retrofit becomes a soft-fall padding hazard the moment an aerial stage precedes it. Transparent omission preserves the existing meaning.
- **Two distinct `Dodge` Movement Stages (`Dodge (ground)` with `footing: "ground"` and `Dodge (air)` with `footing: "air"`), sidebar groups them under one label.** Considered seriously. Rejected for the same Dodge-is-transparent reason — forcing two stages would require the sidebar to resolve which to inject based on current footing (couples UI to tracker state), or the user to manually pick (UX worse than transparent passthrough).
- **Per-character footing throughout (no team-level state).** Rejected. Scenario: charA plays a launch stage and goes aerial, swaps out to charB, charB plays a land stage, charB swaps to charC, charC plays grounded rotations for 10 seconds, charA re-enters via a normal cast. Under per-character, `charA.footing` is still `air` (last on-field write) — charA re-enters aerial after 10 seconds off-field. In-game wrong. The team-global value carries the "what the player has actually been seeing on the active character recently" semantic, which is what a re-entering off-field character (outside any trailing window) physically should be reading.
- **Team-global footing only (no per-character snapshot).** Rejected. The aerial-swap-cancel case fails: after charA's launch swap-variant commits off-field and charB plays a grounded stage, team footing would be `ground` only; charA's swap-back within their own trailing window would re-enter at `ground` despite still being aerial in-game. The snapshot is the minimum extra state that captures the in-game behavior.
- **Eager commit at stage resolution (4-string encoding: `"ground" | "air" | "launch" | "land"`).** Rejected. Collapses two distinct frames — the player's commit point and the animation's actual transition point — into one. In-game, a sufficiently early swap-cancel of a launch stage does **not** reach the point-of-no-return frame, so the launch never happens. The 4-string model has no way to express this; the frame-bearing transition object makes the point-of-no-return explicit in authoring.
- **Two fields on Stage (`footing` + `footingFrame: number`).** Rejected. The frame is meaningful **only** for transitions; encoding it as a separate field invites authoring sustained `"ground"`/`"air"` with a stray `footingFrame: 0` and forces every reader to special-case "is this field meaningful right now?" The discriminated union puts the frame where it has meaning and nowhere else.
- **Always-object form (`{ state: "ground" | "air" } | { transition: "launch" | "land"; at: number }`).** Rejected. Uniform shape, but `{ state: "ground" }` is ceremony around what used to be `"ground"`. Every existing stage gets noisier for no behavioral gain.
- **String union with optional `at` defaulting to "end of advance" (`footing?: "ground" | "air" | { kind: "launch" | "land"; at?: number }`).** Rejected. Most migration-friendly, but hides the load-bearing decision behind a default — the `at?: number` optional re-introduces the eager-commit bug, just with extra steps.
- **Eager commit at swap time when the launch frame is "close enough" to `swap.actionTime`.** Rejected. The whole motivation for the mid-stage commit frame is modeling the point-of-no-return correctly; an eager-if-close-enough heuristic re-introduces the eager-commit bug under a fuzzy threshold. Authors who want a swap-cancel to feel like a launch-swap can author `swap.actionTime ≥ footing.launch`.
- **Discriminated event union for the Trailing Window (`TrailingEvent = { kind: "hit" } | { kind: "footing" }`).** Rejected. A stage has at most one launch/land frame, so a character has at most one pending footing event per in-flight stage — not a list. The union shape would let the trailing window theoretically hold multiple conflicting footing events for one character (nonsense the type system would let you write), and forces every consumer that walks events to grow a `switch`. The struct shape (`TrailingEntry { hits, pendingFooting? }`) encodes the cardinality at the type level and leaves hit-processing code bit-exact.
- **Make `FootingTracker` aware of swap events (consumes `swapIn`/`swapOut` from `OnFieldTracker`).** Rejected. Coupling the two trackers creates an event-ordering dependency the existing engine doesn't have. The on-field check at footing-event dispatch reads `OnFieldTracker` at call-site (engine-layer), but the two trackers themselves remain siblings with no cross-references.
- **Engine-runtime assertion of footing validity (defense-in-depth alongside `validateTimeline`).** Rejected. No other Stage-level validity rule (`requiresStageId`, `comboAllows`) is engine-asserted; the precedent is that the engine trusts the validator. Doubling the rule surface buys no end-user value and creates two places to keep in sync.
- **Air→ground as a hard error (require explicit `land` Movement Stage to bridge).** Rejected. The game itself auto-recovers from aerial state via gravity — there's no "Land button" the player presses; they just fall. Modeling this as a hard error would force every aerial-to-grounded sequence to author a `land` stage that doesn't correspond to any real player input, polluting timelines for no analytic gain. Soft-with-padding matches the actual mechanic.
- **Ground→air as a soft warning with auto-inserted launch padding.** Rejected for the symmetric reason — there is no gravity equivalent that lifts the character off the ground without input. Going aerial requires an explicit `Jump` or character-specific launch move; silently padding the transition would invent frames the player never spent.
- **Attach `fall` padding to the previous entry's `advance` (symmetric with `pad`).** Rejected. The cross-character case is load-bearing: charA aerial swap-variant → charB grounded stage. `pad` on the previous entry would put a `+0.35s fall` on charA's row, but charA isn't the one falling — charB is. The current-entry attribution matches the actual physical event the padding models.
- **Fall padding subject to `variantFloor`.** Rejected. `variantFloor` models the physical minimum input duration a player can execute — a finger-on-button constraint. Fall padding models gravity — a world-state constraint. They are orthogonal physical phenomena and the floor's max-semantics would let one mask the other, producing under-padded sequences whenever both apply. Both should accumulate.
- **`fallFrames` per-character (heavier characters fall slower).** Rejected as overmodeling. The sim doesn't track height, mass, or any other variable that would meaningfully differentiate character fall speeds, and the in-game variance is small enough that a single global default captures the phenomenon well enough to drive optimization decisions. If a future character genuinely needs a different fall constant, the field can be added on Character as an override without changing the global default.

## Consequences

- `Stage` gains an optional `footing?: "ground" | "air" | { launch: number } | { land: number }` field. Existing stages without `footing` are treated as transparent — no migration is needed for non-aerial content.
- `src/data/movement.ts`: `Jump` is `{ launch: <action-frame> }`; `Dodge` omits `footing`. No other Movement Stage fields change.
- `Settings` becomes `{ reactionDelay; swapFrames; variantFloor; fallFrames }`; default `fallFrames: 21`. The `useLocalStorage` merge path handles older saved settings missing the field via the default. A settings-UI control surfaces the new value alongside the existing three.
- A new `FootingTracker` module lives in `src/lib/engine/footing-tracker.ts`, mirroring `on-field-tracker.ts`'s shape: single team-footing value, mutation-versioned, plus a `Map<number, "ground" | "air">` snapshot dict. Public surface: `current()`, `setTeam(footing)`, `snapshotFor(characterId, footing)`, `consumeSnapshot(characterId): "ground" | "air" | null`, `clearSnapshot(characterId)`. The engine composes it alongside the existing collaborators on `BuffEngine`.
- The Trailing Window state (per [ADR-0018](0018-swap-variant-as-trailing-window.md)) becomes `ReadonlyMap<characterId, TrailingEntry>` where `TrailingEntry = { hits: readonly TrailingHit[]; pendingFooting?: { atFrame: number; exitFooting: "ground" | "air" } }`. `scheduleStage` populates `pendingFooting` when the stage carries `{ launch }` / `{ land }` and the variant's resolved `advance` does not pre-empt the commit frame. `onEntryArrival`'s cancel-capable drop and non-cancel-capable pad both consult `pendingFooting.atFrame` alongside hit frames.
- An engine-layer dispatch helper consumes the firing footing event and routes by on-field check: owner on-field → `setTeam(exitFooting)` + `clearSnapshot(ownerId)`; owner off-field → `snapshotFor(ownerId, exitFooting)` only. The check reads `OnFieldTracker` at call-site; the two trackers do not reference each other.
- Swap-in path enforces the on-field invariant: if `consumeSnapshot(charId)` returns non-null, the engine calls `setTeam(snap)` before the entry's footing validity check, so footing-transparent stages after re-entry (e.g. Dodge, or any Basic Attack with no `footing` field) can't leak stale team footing forward. Fall padding then reads `current()` against the incoming stage's required-entry footing.
- `validateTimeline` gains a footing-validity pass. Walks the entry sequence, maintaining a current-footing cursor and a snapshot dict (the trailing-window state is computable from `swap`-variant entries plus their pending hits and pending footing events). Emits row errors for hard cases (`ground→air`, `air→{ launch }`, `ground→{ land }`) and surfaces fall padding as a row annotation (not an error). Exit-footing inference is variant-aware: cancel/instantCancel exit equals the transition target only when the authored commit frame is `≤ resolved variant advance`; otherwise exit equals entry footing. `swap` variants' cursor exit always equals entry footing.
- `Padding Delay`'s tooltip gains a `fall` component. The merged `+0.Xs` suffix on Timeline rows and Simulation Log entries already aggregates components; `fall` participates additively. The mutex (`react` vs. `floor` — only the max shows) is unchanged; `pad` and `fall` both add on top.
- New domain terms enter the project glossary under Core simulation: **Footing**, **FootingTracker**, **Fall Frames**. **Padding Delay** and **Trailing Window** entries are amended with the new component and pending-footing rules respectively. **Movement Stage** is amended to note `Jump`'s `{ launch }` assignment and `Dodge`'s transparent omission.
- The decision is conservative on engine-event surface area: footing transitions do not fire `tookOff` / `landed` events. No currently authored buff triggers on the transition itself (buffs that care about air do so via skill-type / stage-id filters on the already-tagged air stage). A future ADR can introduce events when the first buff needs them.

## Amendment — Snapshot removed; footing is pure team state, deferred commits land at re-entry

The per-character footing snapshot is removed. `FootingTracker` keeps **only** the single team-footing value. The `snapshots_` dict, `snapshotFor` / `consumeSnapshot` / `clearSnapshot`, and the `FootingModule` wrappers `snapshotTrailing` / `promoteOnSwapIn`'s consume step / `clearTrailingSnapshot` all go away. The off-field dispatch branch specified in the original Consequences ("owner off-field → `snapshotFor(ownerId, exitFooting)` only") is dropped — it was specified but never wired: the only producer ever built (`simulation.ts` → `snapshotTrailing`) fires at the owner's **re-entry**, and the only consumer (`promoteOnSwapIn`) fires in the **same loop iteration**. The dict was a per-character store whose entries never spanned a tick.

**Why the "Team-global footing only" rejection above no longer holds.** That option was rejected (Considered Options) on the premise that charA's launch commit _fires while charA is off-field_, so without a snapshot the `air` is lost and charA re-enters at `ground`. The as-built engine never fires the commit off-field. The deferred launch/land lives in the **Trailing Window's `pendingFooting`** — which this same ADR introduced and which is **already per-character** (`Map<characterId, TrailingEntry>`). The `exitFooting` is carried there, untouched, the entire time charA is off-field. The snapshot dict was a redundant second per-character copy of a fact the trailing window already holds. The original rejection conflated "fire the commit off-field" (would lose the value) with "fire the commit at re-entry from `pendingFooting`" (the value is right there) — only the former needs the snapshot, and the former was never built.

**Why team-only is correct, not just sufficient.** Footing is read **only for the on-field character** (fall checks, stage entry-footing validation); an off-field character's footing is never queried. So a character's vertical divergence never needs to sit in footing state while they are off-field — it is parked in their trailing entry. At the one frame it becomes relevant — that character's re-entry — they are on-field again, so "their footing" _is_ team footing, and the surviving commit calls `setTeam(exitFooting)` directly. The on-field invariant is preserved by construction rather than by a promote-from-snapshot step.

**Resulting shape.**

- `FootingTracker`: `current()`, `setTeam(footing)`, `mutationVersion()`. No snapshot surface.
- `FootingModule`: `applyStageFooting` (within-`stageDuration` commits, unchanged); a `commit(exitFooting)` the Trailing Window calls when a deferred commit survives; `current()`. `promoteOnSwapIn` collapses to a plain `current()` read.
- Dependency direction is one-way and acyclic: **Trailing Window → Footing** (it calls `setTeam`/`commit` at fire time). Footing knows nothing about swaps or trailing tails. This is the inverse of the deleted snapshot indirection, which existed to _avoid_ drawing this one-way edge — not to break a cycle (there was none).
- `pendingFooting` on `TrailingEntry` is unchanged; it remains the sole per-character carrier of an in-flight launch/land. Its cancel-capable drop / non-cancel-capable pad rules (this ADR's "Trailing Window event scheduling" section) are unchanged.

**Scope deliberately cut.** This removes the generality of applying a footing commit _while its owner is off-field_. That generality is only needed if a future feature **reads an off-field character's footing at an arbitrary frame** — the forward-looking off-field-damage projection mentioned in the `animationFrames` glossary entry is the candidate. If that feature lands and genuinely needs per-character footing reads, it reopens this in a new ADR and re-introduces per-character state then — paid for by a real consumer, not speculatively.

## Amendment — Per-character footing on the event stream; In-trailing characters carry live footing

The consumer the "snapshot removed" amendment deferred to has arrived: characters that **act while off-field** — the [[In-trailing]] state, where a swapped-out character's last stage is still resolving on the one frame-ordered event stream (see [ADR-0028](0028-emithits-as-frame-honest-worklist.md)'s endgame). Per-character footing is re-introduced, but realized on that stream rather than via the original parallel snapshot dict.

**Deployment State.** A character is in exactly one of: **On-field** (deployed/controlled), **In-trailing** (swapped out, its last stage's Trailing Window still open — duration = the stage's `actionTime`), or **In-reserve** (window passed; inert). Footing is _live_ while On-field or In-trailing; an In-reserve character is `ground`.

**Two footing values on `FootingTracker`:**

1. **Team footing** — the field's vertical state = the On-field character's footing. A _fresh_ swap-in inherits it (swapping while airborne keeps the incoming character airborne — the case the "Team-global is wrong" rejection above was really protecting, and it still holds). On-field launch/land commits set it via `applyStageFooting`.
2. **Carried footing** — `Map<characterId, "ground" | "air">`, a per-character override. Set by a launch/land commit firing while the owner is In-trailing (off-field), and by the window-end reset. Consumed at the owner's next swap-in: if present it is used **and promoted to team footing**; otherwise team footing is inherited.

**Footing commits are stream events (instant, no easing).** A swap stage whose launch/land commit frame falls past its `advance` enqueues a footing commit at that frame onto the event stream; on resolve it sets the owner's **carried** footing (the owner is off-field, so team footing is untouched). It obeys the same drop/pad as trailing hits: a same-character cancel-capable re-entry before the commit frame tombstones it (the launch never happened); a non-cancel re-entry pads past it so it fires.

**In-trailing → In-reserve reset.** An airborne In-trailing character returns to `ground` when its Trailing Window passes without a swap-back — a **window-end reset** stream event at `stageStart + actionTime` setting carried `ground`. A re-entry _within_ the window cancels it (carried `air` survives → the swap-back enters airborne and pays fall); a re-entry _at or after_ window-end lets it fire first (the character is benched to `ground` before it takes the field again). Carried footing is read only at the owner's own swap-in, and every entry drains the stream to its arrival frame before reading, so the lazy reset is never observed stale.

**Why this reverses "snapshot removed" without restoring its mechanism.** The prior amendment was right that the _old snapshot dict_ was redundant — its producer fired at re-entry, never off-field. The new design genuinely fires commits off-field (the generality that amendment cut), but via the unified event stream, so carried footing is the live _result_ of stream commits, not a second copy of `pendingFooting`. `TrailingEntry.pendingFooting` itself dissolves: the Trailing Window's per-character map is gone (ADR-0028), and footing commits/resets are first-class stream members alongside trailing hits.

**Resulting shape.**

- `FootingTracker`: `teamFooting()`, `setTeam(f)`, `carriedFor(id)`, `setCarried(id, f)`, `takeCarried(id)`, `mutationVersion()`. The single-team-value surface is replaced by team + a carried map.
- `FootingModule`: `team()`, `applyStageFooting(footing, stageDuration)` (on-field, unchanged), `commitFor(id, exitFooting)` (a stream commit/reset sets the owner's carried footing), `resolveEntry(id)` (the swap-in read: carried override if any — promoted to team and consumed — else inherited team footing).
- The simulation enqueues the footing commit and the window-end reset onto the pending stream; `resolveArrival` tombstones the commit on a cancel-capable collision, cancels the reset on a within-window re-entry, and pads on a non-cancel collision; `drainPending` applies both via `commitFor`.

**Consequence on the swap-back-pays-fall feature.** The within-window case is unchanged (re-entry before `actionTime` enters airborne and pays fall). A swap-back _after_ `actionTime` now benches to `ground` (no fall) — the deliberately corrected model: a character whose trailing window has passed has "landed in the background." Tests whose intervening teammate action outlasts the launcher's `actionTime` were updated from "pays fall" to "no fall" accordingly.
