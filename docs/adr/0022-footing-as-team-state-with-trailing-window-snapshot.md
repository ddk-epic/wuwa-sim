# Footing as team-level state with per-character trailing-window snapshot override

Stages gain an optional `footing: "ground" | "air" | "launch" | "land"` field encoding both required entry footing and exit footing of the team's vertical-position state. The engine tracks current team footing as a single value, updated by each resolved footing-bearing Stage to that stage's exit. A new `FootingTracker` collaborator owns this state — sibling to `OnFieldTracker`, same single-value-with-mutation-version shape, but entirely orthogonal: swap inference and footing inference do not read each other. `Jump` is assigned `footing: "launch"` (ground→air); `Dodge` omits `footing` entirely and is footing-transparent. Different-footing variants of the same conceptual move (e.g. a stage that exists in both a ground and an aerial form) are authored as separate fundamental Stages rather than a polymorphic stage with footing parameters — they almost always carry different Damage Entries anyway.

**Validity (timeline-authoring-only).** The four mismatch cases:

| current team footing | stage requires                        | result                                  |
| -------------------- | ------------------------------------- | --------------------------------------- |
| `air`                | `ground` (or implied entry of `land`) | soft — `fall` padding fires (see below) |
| `ground`             | `air` (or implied entry of `launch`)  | hard error — "launch/jump required"     |
| `air`                | `launch`                              | hard error — "already airborne"         |
| `ground`             | `land`                                | hard error — "nothing to land from"     |

The asymmetry is physical: gravity is free (falls auto-pad), launches cost an animation (must be explicit). Validation lives in `validateTimeline` only; the engine trusts valid timelines. This parallels how `requiresStageId` and `comboAllows` are already validator-only.

**Fall padding.** A new global player-level setting `fallFrames` (default 21 ≈ 0.35s @ 60fps) joins `reactionDelay`, `swapFrames`, and `variantFloor` in `wuwa.settings`. It surfaces as the `fall` component of Padding Delay — additive (does not compete with the `react`/`floor` max), not subject to `variantFloor` (input-duration minimums are orthogonal to gravity), and **attaches to the current entry's startup delay**, not the previous entry's `advance` — a deliberate divergence from `pad`'s "extend previous entry" convention. The reasoning: `pad` exists because A's trailing hits are still running so A's animation legitimately takes longer; `fall` exists because B's character is mid-air when B wants to start, so the delay is B's startup. The current-entry attribution keeps the UI legible in cross-character cases (Verina's ground stage following an aerial Jiyan swap pays the `fall` — showing it on Verina's row matches her landing animation, where showing it on Jiyan's swap stage would not).

**Team-global with per-character trailing-window override.** Two pieces of state live on `FootingTracker`:

1. **Team footing** — single value (`ground | air`), updated by every resolved footing-bearing stage to its exit footing. Read at every swap-in to determine the incoming character's starting footing.
2. **Per-character trailing-window footing snapshot** — `{characterId → snapshot}` dict, populated when a character swaps out via a `swap` variant while their trailing window is active, cleared when that window expires or the character re-enters.

On swap-in, the incoming character's first stage validates against team footing — **unless** that character has a live snapshot, in which case the snapshot overrides team footing only for that re-entry's validity check. After the re-entered stage resolves, team footing follows its exit value as normal.

This composition resolves a real tension: footing has to be team-global so that "aerial swap to a different character lands them aerial too" (the obvious player-mental-model behavior — Verina inherits Jiyan's airborne state when Jiyan swap-cancels mid-jump). But it also has to remember that the swapped-out character is still in their own trailing window, so when **that same character** re-enters during their window, they're still aerial — even if the team has since transitioned through `ground` via the other character. The snapshot is the bookkeeping minimum that captures this without making the entire system per-character.

**Worked example.** charA does an aerial swap-variant stage; team footing → `air`, charA snapshot → `air`. charB swaps in inheriting team `air`; charB's first stage is grounded, fall padding fires on that stage, team footing → `ground` (charA snapshot still `air` — their window is still open). charA swap-back occurs _within_ their own trailing window: snapshot `air` exists and overrides team `ground`, so charA re-enters at `air`. Without the snapshot override, charA would re-enter at `ground` and the engine would silently lose the fact that the swap-out character was mid-aerial — observably wrong against in-game behavior.

## Considered Options

- **Per-Damage-Entry footing tag, no Stage-level field, no tracker.** Rejected. Validity is a stage-to-stage transition concept (the next stage's legality depends on the previous stage's state), not a per-hit one. Modeling it at the hit level forces every consumer (validator, engine, padding calculator) to scan damage arrays and reconstruct stage state, and the fall-padding question becomes ambiguous (which hit triggers fall?). Stage-level matches the granularity of validity.
- **Per-Skill-Type footing implied by a closed mapping (e.g., aerial-Skill-Type values).** Rejected. Skill Type is already the engine's "what kind of skill produced this" axis (deepens, skill-type bonus, hitLanded filters); overloading it with vertical-position semantics conflates two orthogonal dimensions, and characters with both grounded and aerial Heavy Attacks would need new Skill Types per footing — an explosion the closed taxonomy is designed to prevent.
- **Separate `startsIn` and `endsIn` fields on Stage.** Rejected as redundant authoring surface. The cross-product is only four valid cells (gg, aa, ga, ag), and the two-field form invites nonsense combos at the type level that the four-value union prevents by construction. The collapsed form (`"ground" | "air" | "launch" | "land"`) is the closed taxonomy that mirrors the existing pattern of `VariantKind` and `Skill Type`.
- **Add a fifth value `"any"` or `"preserve"` to denote footing-transparent stages.** Rejected. Every consumer would have to special-case the value (validator, tracker, padding calculator, UI display). Omitting the field entirely — already the convention for optional Stage fields like `variants` and `comboAllows` — communicates the same intent with no extra code paths. A doc note in CONTEXT.md clarifies that omission is intentional, not missing-data.
- **Per-character footing throughout (no team-level state).** Rejected. The "Verina swaps in and inherits Jiyan's aerial state" case is the in-game behavior, and per-character state alone cannot express it — Verina's own footing has no reason to be `air` at swap-in. Per-character is necessary for the trailing-window snapshot but insufficient as the only mechanism.
- **Team-global footing only (no per-character snapshot).** Rejected. The walked example shows the failure mode: after `charA aerial swap-variant → charB lands`, team footing is `ground`; charA's swap-back within their own trailing window would re-enter at `ground` despite still being aerial in-game. The snapshot is the minimum extra state that captures the in-game behavior.
- **Make `FootingTracker` aware of swap events (consumes `swapIn`/`swapOut` from `OnFieldTracker`).** Rejected. Coupling the two trackers creates an event-ordering dependency the existing engine doesn't have. Footing is updated by _stage resolution_, not by _swap events_; the swap-in lookup simply reads current team footing at that moment. Snapshot population happens when a footing-bearing swap-variant stage resolves — also a stage-resolution event, not a tracker-to-tracker message. The two trackers can live as siblings with no cross-references and the model still works.
- **Engine-runtime assertion of footing validity (defense-in-depth alongside `validateTimeline`).** Rejected. No other Stage-level validity rule (`requiresStageId`, `comboAllows`) is engine-asserted; the precedent is that the engine trusts the validator. Doubling the rule surface buys no end-user value and creates two places to keep in sync.
- **Air→ground as a hard error (require explicit `land` Movement Stage to bridge).** Rejected. The game itself auto-recovers from aerial state via gravity — there's no "Land button" the player presses; they just fall. Modeling this as a hard error would force every aerial-to-grounded sequence to author a `land` stage that doesn't correspond to any real player input, polluting timelines for no analytic gain. Soft-with-padding matches the actual mechanic.
- **Ground→air as a soft warning with auto-inserted launch padding.** Rejected for the symmetric reason — there is no gravity equivalent that lifts the character off the ground without input. Going aerial requires an explicit `Jump` or character-specific launch move (which the user authors as a `launch`-footed stage); silently padding the transition would invent frames the player never spent.
- **Two distinct `Dodge` Movement Stages (`Dodge (ground)` with `footing: "ground"` and `Dodge (air)` with `footing: "air"`), sidebar groups them under one label.** Considered seriously. Rejected because Dodge's behavior is genuinely footing-transparent — a single button-press in-game that preserves whichever state you were in. Forcing two stages would require the sidebar to resolve which to inject based on current footing (couples UI to tracker state), or the user to manually pick (UX worse than transparent passthrough). The footing-transparent omission expresses the actual game-mechanic.
- **Attach `fall` padding to the previous entry's `advance` (symmetric with `pad`).** Rejected. The cross-character case is the load-bearing one: charA aerial swap-variant → charB grounded stage. `pad` on the previous entry would put a `+0.35s fall` on charA's row (the previous entry), but charA isn't the one falling — charB is. The current-entry attribution (`+0.35s fall` on charB's row) matches the actual physical event the padding models. Engineering-wise, Padding Delay supports both modes (some sources extend previous, some delay current); the cost is one boolean per source, paid once.
- **Fall padding subject to `variantFloor`.** Rejected. `variantFloor` models the physical minimum input duration a player can execute — a finger-on-button constraint. Fall padding models gravity — a world-state constraint. They are orthogonal physical phenomena and the floor's max-semantics would let one mask the other, producing under-padded sequences whenever both apply. Both should accumulate.
- **`fallFrames` per-character (heavier characters fall slower).** Rejected as overmodeling. The sim doesn't track height, mass, or any other variable that would meaningfully differentiate character fall speeds, and the in-game variance is small enough that a single global default captures the phenomenon well enough to drive optimization decisions. If a future character genuinely needs a different fall constant, the field can be added on Character as an override without changing the global default.

## Consequences

- `Stage` gains an optional `footing?: "ground" | "air" | "launch" | "land"` field. Existing stages without `footing` are treated as transparent — no migration is needed for non-aerial content.
- `src/data/movement.ts` updates: `Jump` becomes `footing: "launch"`; `Dodge` omits `footing`. No other Movement Stage fields change.
- `Settings` becomes `{ reactionDelay; swapFrames; variantFloor; fallFrames }`; default `fallFrames: 21`. The `useLocalStorage` merge path handles older saved settings missing the field via the default. A settings-UI control surfaces the new value alongside the existing three.
- A new `FootingTracker` module lives in `src/lib/engine/footing-tracker.ts`, mirroring `on-field-tracker.ts`'s shape: single team-footing value, mutation-versioned, plus a `Map<number, "ground" | "air">` snapshot dict. Public surface: `current()`, `setCurrent(footing)`, `snapshotOnSwapOut(characterId, footing)`, `consumeSnapshot(characterId): "ground" | "air" | null`, `clearSnapshot(characterId)`. The engine composes it alongside the existing collaborators on `BuffEngine`.
- `validateTimeline` gains a footing-validity pass. Walks the entry sequence, maintaining a current-footing cursor and a snapshot dict (the trailing-window state is already computable from `swap`-variant entries plus their pending hits). Emits row errors for hard cases (`ground→air`, `air→launch`, `ground→land`) and surfaces fall padding as a row annotation (not an error). Snapshot consumption follows the same rule as the engine.
- `Padding Delay`'s tooltip gains a `fall` component. The merged `+0.Xs` suffix on Timeline rows and Simulation Log entries already aggregates components; `fall` participates additively. The mutex (`react` vs. `floor` — only the max shows) is unchanged; `pad` and `fall` both add on top.
- The Trailing Window module (`src/lib/stage/trailing-window.ts`, per ADR-0018 amendment) gains a footing-snapshot hook: when a `swap`-variant stage with a footing exit value resolves, the engine records `(characterId, exitFooting)` on the tracker. When the window closes (via re-entry or natural expiry), the snapshot is cleared. The trailing-window state machine itself remains pure — the engine wires the snapshot side-effect on top.
- New domain terms enter the project glossary under Core simulation: **Footing**, **FootingTracker**, **Fall Frames**. **Padding Delay** and **Trailing Window** entries are amended with the new component and snapshot rules respectively. **Movement Stage** is amended to note `Jump`'s `launch` assignment and `Dodge`'s transparent omission.
- The decision is conservative on engine-event surface area: footing transitions do not fire `tookOff` / `landed` events. No currently authored buff triggers on the transition itself (buffs that care about air do so via skill-type / stage-id filters on the already-tagged air stage). A future ADR can introduce events when the first buff needs them.
