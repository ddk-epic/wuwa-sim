# Movement actions as first-class Stages, not Stage Variants

`Dodge` and `Jump` — the universal player movement actions — are modeled as **first-class Stages** that the user inserts into the Timeline, not as Stage Variants of the preceding stage. The Stages are defined centrally in `src/data/movement.ts` (`DODGE_ACTION_TIME = 21`, `JUMP_ACTION_TIME = 18`, both at 60fps) and injected into every `EnrichedCharacter.skills` at enrichment time. They carry empty `damage`, `type: "Movement"` (a new member of both `SkillCategory` and `SkillType`), and reach the engine as ordinary Timeline Entries with Stage IDs `Dodge::_` and `Jump::_`. At dispatch the engine **bypasses the Phase Pipeline** keyed on `stage.type === "Movement"`: an Action Event is logged for visibility, `tickToFrame` runs so duration-based expirations advance, but no `skillCast` event fires and no resource deltas, cooldown rolls, or `hitLanded` events emit. Reaction Delay does not pad their `actionTime` — the authored values are raw.

Cancellation of the _preceding_ stage remains the responsibility of that stage's own Stage Variant (`cancel` / `instantCancel` from ADR-0008). Movement Stages do not retroactively truncate prior stages; they only consume frames that the player would otherwise spend recovering from animation. To preserve combo continuity through a movement action, the _following_ stage's `requiresStageId` gate is widened with a new optional field `comboAllows: readonly ("Dodge" | "Jump")[]`, type-locked through a discriminated union so it is only legal in the presence of `requiresStageId`. `validate-timeline`'s previous-stage walk skips backwards past entries whose stage's `skill.name` is in `comboAllows` before comparing Stage IDs. The default — field omitted — is **opaque**: any movement entry resets the chain. The `SkillCatalog` UI surfaces Dodge and Jump at the bottom of the focused character's stage list, Dodge first, sharing a `MOVE` `typeLabel`.

## Considered Options

- **Movement as new `VariantKind` values (`dodgeCancel` / `jumpCancel`) on the prior stage.** Rejected. Variants in ADR-0008 are _truncations_ of the stage they belong to — they reduce that stage's effective duration and filter its Damage Entries by `actionFrame`. A real dodge in WuWa has its _own_ animation that consumes frames _after_ the cancel point; it is not just an early-exit of the prior stage. Modeling it as a Variant would either (a) lose the post-cancel frames entirely, producing systematically optimistic DPS, or (b) require Variant `actionTime` to mean two different things (truncation-point on `cancel`, truncation-plus-padding on `dodgeCancel`), which breaks the clean filter rule that ADR-0008 was built around. First-class Stages keep both mechanics orthogonal.
- **A single shared synthetic "Movement skill" referenced by every character.** Rejected. The picker would need a second source merged into `getFocusedStageCatalog` alongside echo stages and character stages, the `skill.id` ↔ `character` binding assumed throughout the codebase would gain an exception, and Stage IDs would become character-agnostic — actually a downside for a future per-character override of dodge frames if the "generic hardcoded" decision is ever revisited. Enrichment-time injection into each character's own `skills` array reuses every existing code path.
- **Per-character authored Movement stages in JSON.** Rejected for the same reason ADR-0009 rejected explicit per-stage IDs: 15+ characters of identical busywork with no leverage. A central constants file is one edit site if the meta or measurement changes; per-character authoring multiplies that by the roster size with no current payoff.
- **Empty `damage[]` alone as the phase-bypass signal.** Rejected after checking `buff-engine.onEvent`: a no-damage stage today still fires the `skillCast` arm, which applies concerto, gates resonance cost, and dispatches `skillCast`-triggered buffs. Relying on emptiness silently routes Movement through resource-altering code paths. An explicit `stage.type === "Movement"` check is the load-bearing signal.
- **Per-stage `phasesless: true` flag** instead of type-based bypass. Rejected as more general than the problem: there is no plausible future where one Movement stage participates in phases and another does not. Coupling the bypass to the type matches the intrinsic property and avoids a flag that would always be set the same way per category.
- **`comboAllows` default transparent (omitted = all Movement skips).** Rejected. Failure modes are asymmetric: opaque-default failures are loud (red row through `validate-timeline`'s existing affordance, user re-authors); transparent-default failures are silent (the gate validates green through a chain the author never intended). ADR-0009 made the same choice on Stage IDs for the same reason — loud failure on rare events beats silent failure on common ones.
- **`comboAllows` as free-floating field (not type-locked to `requiresStageId`).** Rejected. The field is dead data on stages with no gate; allowing it there invites mis-authoring that runtime validators would have to catch. The discriminated-union shape makes "this field exists only when `requiresStageId` is set" a compile-time invariant, matching ADR-0008's preference for closed enums over runtime warnings.
- **Adding Reaction Delay to Movement `actionTime`.** Rejected. Reaction Delay in ADR-0008 models the player _reacting_ to an ongoing animation to decide to cancel it. A Movement Stage in an authored rotation is a _committed_ input — there is no reaction lag to model. Adding the delay would silently inflate the authored 21 / 18 frame values away from the numbers the data author chose.
- **Fire `skillCast` for Movement to future-proof Dodge Counter buffs.** Rejected. Dodge Counter is explicitly out of scope and may never ship. A trigger that fired today would be a no-op surface no buff filters on, but resource-side handling inside `onEvent` (concerto floor, resonance gating, cooldown roll) is **not** trigger-filtered — it runs unconditionally on `skillCast`. Shipping the event surface without the bypass corrupts resource state. The right time to add the event is when the first consumer lands; until then, the bypass is the safe default.
- **Dedicated picker section above echo stages (its own divider, top of the column).** Rejected. Movement is the lowest-frequency click target in a typical authoring session; promoting it to the most prominent position penalizes the high-frequency rows. Bottom-of-character-stages keeps the visual rhythm of the existing two-zone layout intact and makes Movement findable by consistent position.

## Consequences

- `SkillCategory` and `SkillType` each gain a `"Movement"` member. `STAGE_TYPE_LABELS["Movement"] = "MOVE"`. `categoryToSkillType` maps `"Movement"` → `"Movement"` (no roll-up to Basic / Heavy).
- `src/data/movement.ts` exports the two stage definitions and their constants. The catalog enrichment site appends `Dodge` and `Jump` (in that order) to every `EnrichedCharacter.skills` before deriving Stage IDs, so `Dodge::_` and `Jump::_` exist for every character without any per-character authoring.
- `buff-engine.onEvent` short-circuits when the resolved stage's `type` is `"Movement"`: emit the Action Event, advance `tickToFrame`, return. No `skillCast` dispatch, no concerto / resonance handling, no cooldown roll.
- `EnrichedSkillAttribute` gains `comboAllows?: readonly ("Dodge" | "Jump")[]`, typed via a discriminated union with `requiresStageId` so the compiler rejects `comboAllows` on stages without a gate. `validate-timeline`'s previous-stage walk skips backwards past entries whose stage's `skill.name` matches a member of `comboAllows` before comparing against `requiresStageId`. A non-Movement intervening entry always breaks the chain regardless of `comboAllows`.
- The closed-enum value set `("Dodge" | "Jump")` is the same set used by the future `categoryToSkillType` extension; adding a third Movement subtype is a single fan-out across the type union, the constants file, and any authored `comboAllows`.
- `SkillCatalog` renders Movement rows at the bottom of the character-stages list, Dodge above Jump. No new divider — they share the existing character-stage zone. The injected skills carry `hidden: false` and a stage `name` distinct from the empty-string sentinel so the existing `stage.name !== "" && !stage.hidden` filter passes.
- `requiresStageId` reachability now has two skip cases stacked: variants of the underlying stage (ADR-0008) and Movement entries listed in `comboAllows` (this ADR). The walk applies `comboAllows` skipping; variant equivalence applies after the walk lands on a real prior stage.
- Saved timelines from before this change are unaffected — no existing entries reference Movement Stage IDs, and the new `comboAllows` field is optional on the gate side. New saved timelines that reference `Dodge::_` / `Jump::_` are stable across renames because the underlying skill names are central constants, not per-character authored strings.
- Reaction Delay continues to apply only to Stage Variants per ADR-0008. The Movement bypass is an additional orthogonal axis: a stage either runs the full pipeline (default), runs a truncated pipeline (variant), or runs no pipeline (Movement). The three modes do not compose — a Movement Stage cannot itself declare variants, and no current authoring path produces one.
- Future work to give Jump a "ground vs. air" state machine, or to wire Dodge Counter buffs into a real `skillCast` event, is a non-breaking extension: both layer on top of the Movement type without touching the Stage / Variant / `comboAllows` contracts established here.
