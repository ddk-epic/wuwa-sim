# Skill cooldowns enforced at simulation time as a Padding Delay within 1s of ready, and an invalid diagnostic when cast earlier than that

A skill's cooldown is a timing fact the player cannot violate, so — like Swap-back Cooldown and the windowed prior-stage gate ([ADR-0036](0036-prior-stage-gate-with-windowed-mode-and-mindelay-pad.md)) — it is enforced at **simulation time**, never in the frame-agnostic validator. On each cast the sim computes `remaining = lastCast + cooldown*60 − cursor`, the frames still left on the cooldown, and branches on a single window:

- `remaining ≤ 0` — ready. Free cast, no pad, no finding.
- `0 < remaining ≤ 60` — within 1s of ready. Model the forced micro-wait: a new **Padding Delay** component padding the cast's start by `remaining`, `max`-combined with `swapBack` and the prior-stage gate. No finding — a sub-second slip is a legal near-edge the player can't act faster than.
- `remaining > 60` — the cast is placed well before the skill is up. Do **not** pad: a multi-second silent wait would distort the timeline and bury the authoring mistake. The cast resolves at its authored start and the sim emits an **invalid** `skillOnCooldown` diagnostic carrying `remaining`, surfaced red on the row with text "Skill on cooldown for {remaining} more frames" so the author knows how far to move it.

This is the first mechanic that is _both_ a pad and a finding — `followUpDelay` is pad-only, resource gates are finding-only — and the first **runtime** diagnostic rendered as invalid rather than warning. The `60` is a single constant (no per-skill knob), and the anchor for the next cooldown is the cast's actual `effectiveStart` whether it was padded or flagged.

Two cooldown placements already exist in the data, dropped at compile until now, and they mean different things. **Skill-level** `Skill.cooldown` (Encore's Flaming Woolies, Camellya's Valse 4s / Fervor 25s) is _shared_ across every stage of the skill — casting any stage arms one timer keyed by skill id. **Stage-level** `SkillAttribute.cooldown` (Camellya's Ephemeral 25s, Perennial 25s, two stages of one Forte Circuit) is an _independent_ timer per stage, keyed by stage id. The key follows the declaration site; both are recorded per character in a dedicated `lastCastByCooldownKey` map, and if a skill ever declared both, both floors `max`-combine like every other start-floor. `computeCooldown` returns `{ pad, diagnostic? }` and is called in the `max(...)` wait assembly beside `computeSwapBackPad` and the gate.

Because the overflow case renders red, the engine's diagnostic channel — until now hard-wired yellow — gains a severity. A `Diagnostic` carries an optional `severity` defaulting to `"warning"` (so `footingViolation`, `insufficientEnergy`, the concerto gates are unchanged); invalid-severity diagnostics route into the row's `errors` channel and mark the row red. This opens a runtime path to invalid that the render boundary previously reserved for the structural validator's `invalidRowIds`. An invalid row is still simulated — invalid is styling and channel, not a block — so the engine's advisory invariant (a finding never blocks or reshapes a cast) holds: the over-early cast still fires at its authored frame.

Perennial's "within 15s after Ephemeral" is **out of scope** — it is a window _end_ (too **late** ⇒ unavailable), the `maxDelay` / Clock B that ADR-0036 deferred. Being late is a violation the player can commit, so it is a finding about a different axis, not a cooldown pad; it stays deferred.

## Considered Options

- **Pad every over-cooldown cast, no diagnostic (pure ADR-0036 model).** Rejected. A cooldown is a hard timing fact, so padding is technically correct, but silently absorbing a 25s-CD skill placed 10s early into a 10s wait produces a misleading timeline and hides a real authoring error. The 1s window keeps the "player can't act faster" pad for genuine near-edge slips while surfacing gross misplacements as the mistakes they are.
- **Always diagnose, never pad.** Rejected for the opposite reason: a cast one frame early is not an error, it is a forced wait the player physically experiences. Flagging it red would cry wolf on every tight rotation.
- **Enforce cooldown in `validate-timeline` (structural, pre-sim).** Rejected for the same reason ADR-0036 rejected computing `followUpDelay` there: the validator is deliberately frame-agnostic, and cooldown is a frames-in-seconds fact only the sim's timing model knows. Teaching the validator absolute frames would duplicate that model in a second place to keep in sync.
- **Keep `skillOnCooldown` a yellow warning to avoid touching the diagnostic channel.** Rejected. An over-early cast is an invalid placement, not a soft nudge, and should read like other invalid rows. The change is contained — one optional `severity` field and one merge at the render boundary — and generalizes: resource/footing diagnostics can opt into invalid later if we decide they warrant it.
- **Collapse to one placement — always key per-skill, give Ephemeral/Perennial separate skill ids.** Rejected. The two stages are genuinely one skill (one Forte Circuit, one slot) with independent recast timers; splitting them into synthetic skills to dodge a stage-level key would distort the data model to fit the engine. Supporting both placements costs one extra map key.
- **Per-skill configurable pad window instead of a constant.** Rejected as speculative generality with no authored consumer — the 1s slip tolerance is a property of human input timing, not of any individual skill.

## Consequences

- `Skill.cooldown` and `SkillAttribute.cooldown`, previously dropped at compile, thread into `ResolvedStage` (`skillCooldown` from the skill, `cooldown` from the stage) and onto the cast path. `compile-character` copies them as it does the other stage fields.
- The sim gains `lastCastByCooldownKey: Map<string, number>` keyed `${characterId}:skill:${skillId}` and `${characterId}:stage:${stageId}`, storing `effectiveStart`, most-recent-wins. `computeCooldown` reads it in the `max(...)` wait assembly and stamps it after the cast's start is fixed.
- A new Padding Delay component (the seventh) joins `swapBack` and the prior-stage gate in the wait `max`, surfaced on the Timeline row and Simulation Log like the others.
- `Diagnostic` gains an optional `severity: "invalid" | "warning"` defaulting to `"warning"`. `timeline-render-items` routes invalid-severity diagnostics into `errors` and into the red-styling set merged with `invalidRowIds`; the row-messages catalog gains `skillOnCooldown { actor, remaining, severity }` with text "Skill on cooldown for {remaining} more frames".
- The row-messages.md invariant "the view styles a row red purely from `invalidRowIds` membership" widens: red now also comes from invalid-severity runtime diagnostics. The advisory invariant is unchanged — invalid rows still simulate.
- Camellya's Valse (4s), Fervor (25s), Ephemeral (25s), Perennial (25s) and Encore's Flaming Woolies cooldowns become live. Perennial's 15s post-Ephemeral window (`maxDelay` / Clock B) remains deferred.
- New glossary term: **Skill Cooldown** (the sim-time recast floor, skill- or stage-keyed); **Padding Delay** gains a seventh component.
