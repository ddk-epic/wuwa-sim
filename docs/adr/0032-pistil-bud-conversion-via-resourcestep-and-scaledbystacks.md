# Camellya's pistil→bud→Sweet Dream conversion via a `resourceStep` trigger and a `scaledByStacks` snapshot

Camellya's Crimson Pistil economy is modeled faithfully rather than abstracted, because the user wants the resource bar and the bud-driven damage scaling to be real — and the primitives it needs are general enough to reuse on other characters. Pistils are the **forte** resource (`forteCap: 100`, refilled to 100 by Intro Everblooming and Ephemeral). Consuming attacks drain forte via **negative `DamageEntry.forte`** values (the empirical per-stage consumption divided evenly across the stage's hits); to keep that correct, `resource-accrual.ts` applies Forte Recharge to **gains only** (`forte > 0 ? forte × (1 + FR) : forte`), so negative forte is applied raw — recharge boosts generation, not the size of a cost. Every 10 forte consumed mints one **Crimson Bud** through a new `resourceStep` trigger; the bud count is read into **Sweet Dream**'s `bonusMultiplier` at Ephemeral cast via a new `scaledByStacks` value expression, snapshotted so the frozen value survives the buds' immediate removal. Concerto stays the per-hit API values (which already encode the conversion's concerto), so `resourceStep` emits **only** a bud, never the +4 concerto — emitting both would double-count.

`resourceStep` is sugar over the existing crossing machinery: at bootstrap it registers a threshold at every multiple of `step` and reuses `fireResourceCrossed`'s per-threshold dispatch (buff-engine.ts:843–865), which already fires once per boundary crossed in a single delta — so a hit that drains through two boundaries mints two buds with the remainder carried, no accumulator needed. Boundary semantics: fire for each multiple `m` with `after ≤ m < before` (so a 100→90 drain mints one bud, and refills, being gains, never fire). Even-division of the negative `forte` keeps per-hit increments small (~3.5), so the multi-cross path is rarely exercised and bud timing is fine-grained.

We rejected: (a) **abstracting the economy** (display-only forte + assume-max-buds) — the user judged dynamic bud count load-bearing; (b) **ten brute-force `resourceCrossed` buff defs**, one per multiple of 10 — works today with zero primitives but is copy-paste data that drifts; (c) a **dedicated `forteCost` field** — cleaner per-field semantics, but it proliferates damage-entry fields for a quantity negative `forte` expresses fine once Forte Recharge is defined to scale gains only; the one hit that both _gains and costs_ forte (which a single signed field cannot express) exists in no kit; (d) putting Sweet Dream in the **additive `dmgBonus`/`amp` buckets** — "increase the DMG **Multiplier** by N%" is the standalone `bonusMultiplier` factor per `references/wording-conventions.md`, and the additive buckets would understate it whenever her other DMG Bonus is present; (e) **modeling normal/Blossom Mode as engine state** — attack availability is the user's authoring choice in a timeline-driven sim, so only Budding Mode (the one mode carrying a damage effect) needs to exist.

## Consequences

- **Consumption is negative `DamageEntry.forte`** — `resource-accrual.ts` scales Forte Recharge over gains only (`forte > 0 ? forte × (1 + FR) : forte`), so negative forte applies raw through the same `applyResourceDelta` path as gains; the ledger's floor-at-0 (ADR-0031) means you cannot mint buds from pistils you do not have. Authored as the empirical per-stage total divided evenly across the stage's hits (fractional values are fine). A hit is a _consuming attack_ iff its `forte < 0` — the marker ADR-0033's Energy Regen Multiplier keys off.
- **New trigger `resourceStep`** (`{ event, resource, step, direction: "consumed" | "gained" }`) joins the event taxonomy (ADR-0002): bootstrap expands it to per-multiple thresholds in the trigger index and it rides the existing crossing dispatch. Carryover and multi-cross fall out of the positional thresholds for free.
- **New value expression `scaledByStacks`** (`{ kind, buffId, characterId, per, base, max, snapshot? }`) resolves to `base + per × min(stacks_of(buffId), max)`. With `snapshot: true` it freezes at apply, so Budding Mode's Sweet Dream captures the live Crimson Bud count at Ephemeral cast and the same cast's `removeBuffs` consumes the buds without disturbing the frozen value.
- **Crimson Bud** is a self-buff: `stacking: { max: 10, onRetrigger: "addStackIndependent" }`, `duration: seconds 15`. Each bud carries its own 15s expiry (`BuffInstance.stackExpiries`) and decays FIFO, so the bud count falls one at a time during an idle gap rather than all riding a single shared timer.
- **The `addStack*` family**: `addStackRefresh` (was `addStack`) adds a stack and resets the shared timer; `addStackKeep` (was `addStackKeepTimer`) adds a stack and keeps it; `addStackIndependent` gives each stack its own expiry via `BuffInstance.stackExpiries` (ascending FIFO), pushing `frame + duration` on retrigger, dropping the oldest at `max`, and pruning per-stack in `tickToFrame` so stacks decay one at a time until the instance empties.
- **Budding Mode** is one `permanent` self-buff (no intrinsic timer) ended by `expiresOnSourceSwapOut: true` and a pistil-zero `resourceCrossed { forte, 0, down } → removeBuffs`. It carries the snapshotted Sweet Dream `bonusMultiplier` (`appliesToHits`-scoped to her core attacks) and doubles as the presence flag S3 conditions on. Its absence _is_ normal/Blossom Mode — Sweet Dream cannot leak pre-Ephemeral because the instance does not exist yet.
- **Opt-in trigger-time `condition` gate on self-buffs** — extends the precedent at buff-engine.ts:199 (where `nextOnField`/reaction buffs already gate triggering on `condition`) to stateful self-buffs behind an opt-in flag, so the Crimson Bud buff can be suppressed during Budding Mode ("cannot gain Crimson Buds") even though forte still drains there.
- **`appliesToHits.stageId` now matches by lineage**, reusing the same `stageIdMatches` the trigger matcher uses (lifted from instance-store.ts to stage.ts): a filter id without a `.<hitIndex>` suffix matches every hit of that stage lineage. This lets Sweet Dream scope to ~15 core-attack stages (some with 19–20 hits each) by listing one lineage id per stage instead of enumerating every hit, and unifies a latent inconsistency where triggers matched stageIds by lineage but the Hit Filter required exact equality. Backward-compatible: existing exact-with-`.N` filters (S2/S3) still require an exact match.

## Known limitations to revisit

- ~~**Per-bud 15s expiry is not modeled**~~ — Resolved. Crimson Bud now uses `addStackIndependent`, so each bud carries its own 15s clock via `BuffInstance.stackExpiries` and decays FIFO; a long idle gap mid-build now drops buds one at a time, lowering the snapshotted Sweet Dream at the next Ephemeral. Decay emits `buffStacksChanged` so the log shows the falling count.
- **A single signed `forte` cannot express simultaneous gain + cost on one hit** — accepted because no kit does it; if one ever does, revisit the dedicated-field option (c).
- **Skill-replacement gating is not engine-enforced** — Ephemeral (and later Perennial) are free-to-author Forte Circuit stages; "Resonance Skill is replaced when concerto is full" is not validated, the same resource-agnostic limitation as ADR-0031's Outro gating.
- **S6 Forte Circuit Perennial is deferred** — a second Ephemeral-like replacement; see `tmp/camellya-perennial-handoff.md`. It adds no new primitive, only a damage stage and a second Sweet Dream value variant.
- See ADR-0033 for the companion energy-accrual decision (the Energy Regen Multiplier keyed off this ADR's `forte < 0` marker).
