# Stage ID as the canonical Timeline Entry reference

> The id **encoding** described here (`"<SkillName>::<StageName>"`) is superseded by the lineage format of [ADR-0024](0024-stageid-lineage-with-skill-grouping-and-type-descriptor.md). The structural decisions — one canonical synthetic id, name-based loud-failure rename semantics, the Stage Resolver seam, the shrunken `TimelineEntry` — all stand.

A **Timeline Entry** references the **Stage** it executes by a single string id, the **Stage ID**, derived at enrichment time as `` `${skill.name}::${stage.newName ?? '_'}` `` for character skills and `` `${echo.name}::${stage.newName ?? '_'}` `` for Echo Skills. The id is synthetic — it does not appear in raw character data — and is attached to every enriched stage by the catalog. After this change, `TimelineEntry` is `{ id, characterId, stageId, variantKind? }` and nothing else: `skillName`, `skillType`, `attackType`, and `multiplier` are removed. All four were caches of stage-derived data; with a stable Stage ID, every consumer that needs them resolves the Stage on demand through a single seam, the Stage Resolver (`src/lib/stage/stage.ts`), which exposes `resolveStage(entry, slots, loadouts) → { stage, element, concerto, damage }`, `resolveStageExecution(stage, variantKind, reactionDelay) → { duration, damage }`, and `stageLabel(skillName, newName)` for display. `requiresStageId` on `EnrichedSkillAttribute` — which previously held a rendered label like `"Normal Attack · Stage 1"` and was string-matched against `entry.skillName` — now holds a real Stage ID. Existing localStorage timelines migrate best-effort on load: each entry's old `skillName` is parsed back into a Stage ID by the inverse of the label rule; rows that fail to migrate lose their `stageId` and surface as invalid through the existing `validate-timeline` machinery, which already renders them red and prompts the user to re-pick.

## Considered Options

- **Stable index `${skill.id}.${stageIndex}`.** Rejected: the stage array is authored as an ordered list and we routinely insert and reorder stages during data work. Index-based ids fail silently — every saved timeline pointing at a later stage suddenly references the wrong one with no validation hook. Name-based ids fail loudly: a rename invalidates the row, validate-timeline flags it, the user re-picks. Loud failure on rare events beats silent failure on common ones.
- **Author `stageId` explicitly in character data files.** Rejected for now: every stage gets a hand-written id, every echo too, and the migration of existing data (15+ characters, dozens of stages) is pure busywork with no leverage today. The synthetic name-based scheme can be replaced with explicit ids later if rename churn becomes a real cost; the migration would be a one-time author pass and existing timelines would re-migrate through the same loud-failure path.
- **Keep `skillName` on `TimelineEntry` as a cached display label.** Rejected: it's the original smell — a rendered string used as both a key and a display field, with the rendering rule (`stageLabel`) duplicated across four files because the lookup needed it. The whole point of introducing Stage ID is that display is a render concern, not a stored one. Keeping `skillName` re-creates the drift risk.
- **Keep `attackType` and `multiplier` on `TimelineEntry` as caches.** Rejected: both are pure projections of `stage.damage` (`attackType` via `deriveAttackType`, `multiplier` as a sum), each used in exactly one site. With `resolveStage` already on the hot path of every consumer, the caches buy nothing and add two more fields to keep coherent through every edit.
- **Hard cutover migration (bump schema version, discard old data).** Rejected: best-effort migration is genuinely cheap because old `skillName` values are exactly `stageLabel(skill.name, stage.newName)` — the inverse is mechanical. A user who saved a timeline yesterday should not lose it because we tightened the schema today.
- **Migrate-or-fail-loudly (any failed row throws on load).** Rejected: the common failure case is one stale row from a renamed or deleted stage; the right UX is to mark that row invalid and let the rest load, not to refuse the whole file.
- **One Stage Resolver module per concern (`stage-resolver.ts` + `stage-execution.ts` + `stage-label.ts`).** Rejected: Stage is one concept in `CONTEXT.md`. The recent variant-suffix change (`stageLabel` learning about variants) already showed that label rules, lookup, and execution are coupled; splitting them across files would reproduce the very drift this refactor exists to remove. One `src/lib/stage/stage.ts` is the seam.

## Consequences

- `EnrichedSkillAttribute` gains a synthetic `id: string` field, populated by the catalog at enrichment time. Raw `Skill`/`Stage` types in `src/types/character.ts` (the authored shape) are unchanged.
- `TimelineEntry` shrinks to `{ id, characterId, stageId, variantKind? }`. `skillName`, `skillType`, `attackType`, and `multiplier` are removed. Consumers that previously read these fields call `resolveStage` and project from the result.
- `src/lib/resolve-action-time.ts` is deleted. `resolveActionTime` is absorbed into `resolveStageExecution` inside `src/lib/stage/stage.ts`. The duplicated `stageLabel` and stage-lookup helpers in `simulation-log.ts`, `timeline-summary.ts`, and `validate-timeline.ts` are removed in favor of `resolveStage`.
- `TimelineView` uses `resolveStage` for row display and `stageLabel` from `stage/stage.ts` for the picker. Its inline duplicates are removed.
- `requiresStageId` field semantics tighten: it now holds a real Stage ID. Test fixtures and any existing character data using label-form values must be updated. The cascade logic in `validate-timeline` continues to work — it just compares Stage IDs instead of labels.
- localStorage migration is best-effort and idempotent: on load, any entry without `stageId` derives one from `skillName` by parsing the label format; any entry with `stageId` is left alone. Failed rows lose `stageId` and validate-timeline flags them invalid, surfacing through existing UI affordances.
- Renaming a skill or a stage's `newName` invalidates every saved timeline that references it. This is a deliberate trade against silent index-shift bugs. Renames are deliberate, visible authoring actions; failed-to-resolve rows are user-recoverable through the existing invalid-row UI.
- Future work to author Stage IDs explicitly in data files is a non-breaking follow-up: the `stage.id` field already exists on the enriched type, so swapping the source from "synthesized from name" to "read from data" leaves the rest of the codebase untouched.
