# Post-`actionTime` hits resolve in global frame order for every stage, not only swap variants

`partitionStage` now splits a stage's hits into **immediate** (`actionFrame ≤ stageDuration`) and **trailing** (`actionFrame > stageDuration`) for **all** stages, where it previously did so only for `swap` variants and dumped every hit of a non-swap stage into `immediate`. Trailing hits enqueue onto the frame-ordered `Schedule` instead of resolving synchronously in the entry's processing loop. A non-swap stage's trailing hits enqueue with the **`ignore`** arrival class (uncollidable, always land); `swap` keeps **`residue`** (the cancel/pad window). Footing changes stay swap-only.

The motivating bug: `Flaming Woolies (swap) → Thermal Field → Freezing Thorns → Energetic Welcome` with Encore on Stringmaster. Thermal Field is Encore's Outro — `actionTime 0`, but its DoT hits land at `actionFrame` 90/180/270/360. As a non-swap stage every hit was `immediate`, so the entry loop resolved them synchronously through `processHit → advanceTo(hitFrame)`, monotonically dragging the engine clock to ~f=468 while the timeline cursor stayed at ~f=108. Stringmaster's `Electric Amplification (ATK)` (`addStack`, 5s) was applied at f=0, expiring at f=300. Energetic Welcome — a Resonance Skill at f=103, inside that live window — was processed **after** the outro hits had already ticked the clock past 300, so `applyBuff` could not find the live instance and spawned a **second** one (`[103,403]`) instead of adding a stack. Two overlapping instances where the game shows one at ×2. The buff-timeline lane packer (which owns a lane per buff identity and spills genuine overlaps; [ADR pending / build-buff-timeline-model]) then correctly rendered them in two lanes — surfacing the engine fault as a visible "the buff jumped lanes."

The root cause is that a stage's `actionTime` is when control returns, but its hits can land later; those later hits are **background damage** that must interleave with subsequent actions in true frame order, exactly as `swap` trailing hits already do. Gating the partition on `swap` was the accident — the frame-ordering need is general, not swap-specific. Deferring post-`actionTime` hits to the `Schedule` keeps the engine clock in step with the cursor, so a later cast that resolves to an earlier frame is dispatched **before** the prior stage's tail, and buff lifecycle (stacking, expiry) is evaluated against the clock state that actually held at the cast frame.

The arrival class matters: a non-swap stage's trailing hits are not cancellable residue. An Outro's DoT keeps ticking regardless of what the owner does next, so enqueuing them as `residue` (the first cut of this change) let Energetic Welcome's own cancel-capable re-entry drop three of the four outro hits via `cancelResidue`. Enqueuing them as `ignore` — never invalidated, uncollidable — lands all four **and** lets the follow-up stack.

This is orthogonal to the `independent` flag ([ADR-0008](0008-stage-variants-as-actionframe-truncation.md)). `independent` is consulted only in the `cancel`/`instantCancel` arm of `resolveStageExecution` and decides whether a hit **survives a self-declared truncation**. The partition decides **where a surviving hit resolves** (synchronously vs. the frame-ordered stream). They compose: an `independent` hit that survives a cancel now also defers to the stream rather than advancing the clock, which is strictly more correct.

## Considered Options

- **Mark each Outro/long-tail hit `independent: true` (data fix).** Rejected. `independent` is a different axis — it only governs cancel-truncation survival and is ignored for the no-variant and swap cases, so it changes nothing for an un-cancelled Outro. It also pushes a general ordering-correctness concern onto per-hit authoring, where a forgotten flag silently restores the synchronous-clock bug.
- **Keep the partition swap-only; reconcile in the engine when `applyBuff` is handed a frame behind the clock high-water mark.** Rejected. The engine clock is monotonic by design; retroactive application against a past frame is unsound and would special-case every lifecycle path (stacking, expiry, snapshots) instead of fixing the one ordering fault.
- **Enqueue non-swap trailing hits as `residue` (reuse the swap path verbatim).** Rejected — it drops an Outro's DoT when the owner acts again, because `residue` is the cancel/pad window. Background damage needs `ignore`.
- **Treat the timeline as invalid authoring (Outro then same-character follow-up).** Rejected. The combination is legal — the Outro behaves like a buff emitting a DoT; the swap-out cast leaves Encore In-trailing while a teammate interweaves, and she returns for the follow-up. The ordering fault also generalizes to any character acting before a prior stage's long-tail hits land.

## Consequences

- `partitionStage` ([trailing-window.ts](../../src/lib/trailing-window.ts)) splits `immediate`/`trailing` by `actionFrame ≤ stageDuration` for every `variantKind`; `footingChanges` stay swap-only. The `partitionStage: non-swap partition` unit test is updated from "all hits immediate" to "post-`actionTime` hits defer to trailing."
- The trailing enqueue in [simulation.ts](../../src/lib/simulation.ts) branches arrival on `swap`: `residue` for swap windows, `ignore` (owner-less, uncollidable) for non-swap background hits.
- Resources (energy/concerto) from a post-`actionTime` hit now land at the hit's true frame rather than synchronously inside the casting entry — more frame-honest, with no existing fixture depending on the old timing (full suite green save the one rewritten unit test).
- A regression test (`simulation.buffs.test.ts`) fixes a long-tail hit prematurely expiring a buff a later earlier-framed cast should stack onto: one instance at ×2 stacks, not two overlapping instances.
- Extends the frame-ordered-stream model of [ADR-0028](0028-emithits-as-frame-honest-worklist.md) and the trailing-window machinery of [ADR-0018](0018-swap-variant-as-trailing-window.md) from swap variants to all stages. Independent of the `independent` flag ([ADR-0008](0008-stage-variants-as-actionframe-truncation.md)).
