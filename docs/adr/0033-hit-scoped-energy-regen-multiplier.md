# Per-skill Energy Regen Multiplier via a hit-scoped `energyGainMult`, breaking the hit-agnostic energy-accrual contract

Camellya's Forte Circuit grants "+150% Energy Regen Multiplier" to her consuming attacks and drops it "to 0%" in Budding Mode — a per-skill multiplier on Resonance Energy generation that the per-hit `energy` values (raw API base) do not bake in. Modeling it faithfully forces energy accrual to become **hit-scoped**: gain becomes `entryEnergy × (1 + ER) × (1 + energyGainMult)`, where `energyGainMult` is a buff-contributed bucket read for the specific hit that landed. This reverses the standing contract that energy/forte accrual is hit-agnostic (the `#321` note: ER/FR resolved without hit context). Only the new ERM bucket is hit-scoped; ER and FR stay hit-agnostic. The multiplier is keyed off the **negative-`forte` marker** from ADR-0032 — a hit that carries `forte < 0` _is_ a consuming attack, so the set "attacks that consume pistils" and the set "attacks that get +150% ERM" are one list with one source of truth.

"Reduced to 0%" cannot be expressed as additive accumulation (the bucket sums to an offset, base 0 → ×1; there is no "set" op). We model it with a **negated condition**: Forte contributes `energyGainMult += 1.5` gated `NOT buffActive(budding)`, and Budding Mode contributes `energyGainMult += −1.0`, so the net in Budding is `−1.0` → factor 0, and outside Budding is `+1.5` → factor 2.5. Each buff states its own value and the cancellation emerges, rather than one buff hardcoding the other's magnitude.

We rejected: (a) **`SkillType`-keyed ERM** (the way `skillTypeBonus`/`deepen` are keyed) — too coarse: her consuming attacks are all typed `Basic Attack`, Seedbed recategorizes Heavy Pruning _to_ Basic Attack, and her mid-air/dodge-counter hits are Basic Attack too, so a `SkillType` scope sweeps in hits the +150% list excludes and cannot be reconciled against the prose list without guesswork; the `forte < 0` marker sidesteps that entirely (whatever you author as consuming gets the ERM); (b) **baking the +150% into the per-hit `energy` values** — the user confirmed those are raw API base values, so baking would be unfaithful and would also have no clean way to drop to 0% in Budding; (c) **Budding hardcoding `−2.5`** to cancel base + Forte without a negated condition — works but couples Budding's value to Forte's, breaking if either changes; (d) **a stat `set` op** for the 0% — a larger, riskier engine change than a reusable negated condition, and accumulation-by-set conflicts with the additive model everywhere else.

## Consequences

- **New StatTable bucket `energyGainMult`**, folded into energy accrual as a `× (1 + energyGainMult)` factor. Accrual gains a hit-scoped read for this one quantity; ER/FR remain hit-agnostic, so the `#321` contract is narrowed, not discarded — "energy gain is hit-agnostic _except for an explicit per-skill regen multiplier_."
- **ERM is scoped off the negative-`forte` marker** (`forte < 0`, ADR-0032), not a `SkillType` key. Single source of truth: authoring a hit with negative `forte` declares it a consuming attack and the ERM rides along. Even-division of the consumption means every consuming hit carries the marker, so the +150% lands uniformly with no first-hit-only asymmetry.
- **New `Condition` negation** (`buffActive { …, negate: true }`, or a sibling `buffInactive`) joins the condition taxonomy — reusable wherever an effect must turn off while another buff is present. Here it makes Forte's +150% vanish in Budding so Budding's −1.0 zeroes the factor.
- **Faithful, reusable beyond Camellya** — any future character with a per-skill energy/regen multiplier (or a "no energy in mode X") uses the same bucket + negated condition rather than a bespoke hack.

## Known limitations to revisit

- **ERM keyed off the sign of `forte` is an unusual coupling** — an energy rule reads whether a hit's `forte` is negative rather than a stat bucket. Deliberate (it dissolves the prose-vs-empirical list discrepancy), but a reader will not expect energy semantics to depend on a forte field's sign; documented here so the coupling is intentional, not accidental.
- **`energyGainMult` is additive-only** — no "set to N%" op; the 0% case is achieved by cancellation, which works because we control both contributors. A character whose multiplier is _set_ by an external source (not self-cancelled) would need more.
- **The negated condition only tests buff presence** — it does not yet compose with the other condition kinds (no `AND`/`OR`); sufficient for this kit.
- Companion to ADR-0032 (the conversion mechanic that defines the negative-`forte` marker this ADR keys on).
