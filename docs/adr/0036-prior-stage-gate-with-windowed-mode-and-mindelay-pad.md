# Prior-stage gate generalized to chain + windowed modes via implicit `minDelay`, `comboAllows` retired

`requiresStageId` is renamed to **`requiresPriorStageId`** and generalized from a single positional combo gate into a two-mode gate discriminated by the presence of a new optional sibling, **`minDelay`** (frames). With `minDelay` absent the gate keeps the existing **chain** semantics — the prerequisite stage must be the immediately preceding same-character entry (current `requiresStageId` behavior, ADR-0009/ADR-0024). With `minDelay` present the gate switches to **window** semantics — the prerequisite must have cast earlier on the same character at any distance, and intervening entries (swap-out, teammate entries, the actor's own other actions) do **not** break it; the follow-up simply cannot fire until `minDelay` frames after the prerequisite's cast. The motivating case is Encore's Energetic Welcome, a follow-up to Flaming Woolies castable only ~1.7s (103 frames, Flaming Woolies' last hit `actionFrame`) after the Flaming Woolies cast and remaining available across a swap-out so long as Encore returns in time.

The two checks live in different layers, matching the engine's existing division of labor. **Existence/ordering is validation-time** (`validate-timeline`, which stays frame-agnostic): chain mode keeps its immediately-preceding-must-match walk; window mode does a same-character backward scan over the whole prior timeline for a matching prerequisite cast, hard-erroring if none exists. That guarantees the simulation will find an anchor. **Timing is simulation-time**: the sim records each prerequisite's cast/start frame and, on the follow-up entry, pads its start by `max(0, anchorFrame + minDelay − cursor)` — a new [[Padding Delay]] component max-combined with `swapBack` (both are absolute-frame floors on the same entry's start, not additive costs). The anchor is the prerequisite's **cast frame**, not its last hit frame; because cast-time effects fire even on a swap-cancel (ADR-0018), the anchor is recorded for a swap-cancelled Flaming Woolies too — which is exactly why the In-trailing swap-back case needs no dedicated machinery: by the time the 103-frame `minDelay` elapses, the swap stage's 108-frame Trailing Window has essentially closed anyway. Multiple prior casts anchor to the **most recent** before the follow-up, so re-casting re-arms it. In the common full-cast case (Flaming Woolies advances its full 108 ≥ the 103-frame `minDelay`) the pad is **0** — the gate is free, biting only when the prerequisite was swap-cancelled and the actor returns early, i.e. the real "wait for the woolies" delay.

`minDelay` is **too soon ⇒ pad, never error**: like Swap-back Cooldown, it is a timing fact the player cannot violate, so the engine models the forced wait rather than red-flagging a legal intent. The upper bound (a window _end_ past which the follow-up can no longer be cast — "Clock B") is **deferred**: no `maxDelay` field is reserved on the type until that validation case is actually built.

In the same change **`comboAllows` is retired entirely** — removed from `EnrichedSkillAttribute`, from `validate-timeline`'s skip-walk, and from `CHARACTERS.md`. It was speculative generality from ADR-0015 with zero authored consumers (it appeared only in documentation), and its removal is behavior-neutral: no authored chain declared it, so every existing combo already breaks on any intervening entry including Dodge/Jump. Should a future combo genuinely need to tolerate a mid-chain movement, window mode is a strictly more general expression of "intervening entries don't break the gate" than a movement whitelist ever was.

## Considered Options

- **Explicit `mode: "chain" | "window"` discriminator instead of implicit `minDelay` presence.** Rejected. It would protect two "orthogonal" axes — an immediate gate that _also_ carries a min delay, and a non-breaking gate with _no_ delay — that are exactly as speculative as `comboAllows`: no authored stage wants them, and WuWa's combo model is a max input buffer, not a min delay, making "immediate + minDelay" implausible. Building mode-tag scaffolding to keep those cases legible would repeat the over-generalization this ADR is deleting. The one real subtlety — that `minDelay` presence also flips the gate non-positional — is captured in a type comment and the `CHARACTERS.md` entry.
- **A separate field (`followsStageId` / `requiresPriorStage` object) for the windowed case, leaving `requiresStageId` as-is.** Rejected. Two fields for one concept ("this stage requires a prior stage") drift and force every consumer to check both. Generalizing the single existing field — and migrating the current combo data onto it — keeps one gate with one validator branch.
- **Keep `comboAllows`, only flip its default to always-transparent.** Rejected as half-measure. The field has no consumer and no foreseeable one; flipping its default still leaves the type union, the skip-walk, and the doc surface to maintain. Deleting it removes all of that, and window mode subsumes the capability if it is ever needed.
- **Lose positional combo-breaking — make every gate windowed.** Rejected. `Stage 2 → Resonance Skill → Stage 3` is genuinely illegal in-game (a skill resets the basic chain to Stage 1) and the positional gate models that correctly today; a purely temporal window would validate it green if the intervening action fit inside the window. Chain mode is retained precisely to preserve break-on-non-movement.
- **Compute `minDelay` timing in `validate-timeline` (hard error when too soon).** Rejected. The validator deliberately does not compute absolute frames; teaching it to would duplicate the simulation's timing model in a second place that must be kept in sync. Confining all timing to the sim, where Swap-back Cooldown and Fall Frames already live, keeps one source of truth.
- **Anchor `minDelay` on the prerequisite's last _hit_ frame rather than its cast frame.** Rejected. "After the cast" is the in-game phrasing, and the last-hit frame drifts under swap-cancel (trailing hits land on their own schedule), which would make the anchor depend on variant and footing state. The cast frame is stable and recorded uniformly across full and swap-cancelled executions.
- **Reserve a `maxDelay` / window-end field now.** Rejected for the same YAGNI reason as the explicit mode tag — Clock B is deferred, and an unused field invites the speculative upkeep this ADR is cutting. It is added when the validation case is built.

## Consequences

- `EnrichedSkillAttribute`'s discriminated union loses its `comboAllows` arm. The gated arm becomes `{ requiresPriorStageId: string; minDelay?: number }`; the ungated arm becomes `{ requiresPriorStageId?: never }`. `MovementKind` survives (footing/`Jump` still use it) but no longer appears on the gate.
- All current `requiresStageId` usages (Sanhua Frigid Light; Encore Wooly Attack and Cosmos: Frolicking chains) migrate field-name-only to `requiresPriorStageId` with no `minDelay` — bit-identical chain behavior. Encore's Energetic Welcome gains `requiresPriorStageId` (Flaming Woolies) + `minDelay: 103`.
- `validate-timeline` gains a window-mode branch: when `minDelay` is set, replace the immediately-preceding check with a same-character earlier-anywhere scan; absence of a match is a hard error. The `comboAllows` skip-walk is deleted.
- The simulation records prerequisite cast frames (most-recent-wins, per character + prerequisite stageId) and computes the follow-up pad as a new Padding Delay component, attributed to the follow-up entry and `max`-combined with `swapBack`. Surfaced on the Timeline row and Simulation Log like the other pad components.
- `CHARACTERS.md` Cookbook recipe 3 (multi-stage combo) updates the field name and drops the `comboAllows` mention; a new recipe documents the windowed follow-up (`minDelay`). The Stage-fields and "Rare fields" tables drop `comboAllows` and rename `requiresStageId`.
- Partially supersedes [ADR-0015](0015-movement-as-first-class-stages.md): the `comboAllows` field and its validator walk introduced there are removed. Movement Stages themselves are unaffected.
- New/changed glossary terms: **Prior-Stage Gate** (the generalized `requiresPriorStageId`, with its chain/window modes and `minDelay`); the **`comboAllows`** term is deleted; **Padding Delay** gains a sixth component.
