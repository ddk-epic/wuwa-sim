# Prior-stage gate: array-valued OR gate with chain/window modes, `swap-in` opener sentinel, and `followUpMin`/`followUpMaxDelay`

`requiresStageId` was renamed to **`requiresPriorStageId`** and generalized twice: first from a single positional combo gate into a two-mode gate, then from a single prerequisite into an **array** of alternatives. The authoring field `requiresPriorStage` accepts `string | string[]` and compiles to `requiresPriorStageId: string[]`; the gate is satisfied when **any** listed prerequisite qualifies (OR / any-of). A single string is sugar for a one-element array, so existing single-prerequisite chains are unchanged.

The gate has two modes, discriminated by the presence of a **delay** sibling. With no delay field the gate is **chain** mode: the immediately-preceding same-character entry must **be one of** the listed stages (a skill or dodge that resets the basic chain breaks it). With a delay field present the gate is **window** mode: **any** of the listed stages must have cast earlier on the same character at any distance, and intervening entries (swap-out, teammate entries, the actor's own other actions) do **not** break it.

Two delay fields bound the window, both in frames, both optional, either one flipping the gate to window mode:

- **`followUpMinDelay`** (formerly `followUpDelay`) — a **lower** bound. The follow-up cannot fire until `anchorFrame + followUpMinDelay`; the sim pads its start by `max(0, anchorFrame + followUpMinDelay − cursor)`. The motivating case is Encore's Energetic Welcome, castable only ~1.7s (103 frames) after Flaming Woolies and surviving a swap-out so long as Encore returns in time.
- **`followUpMaxDelay`** — an **upper** bound (the "Clock B" this ADR originally deferred). Past `anchorFrame + followUpMaxDelay` the window has **closed** and the follow-up is no longer castable. The motivating case is Cartethyia's May Tempest Break the Tides, castable within an 8s (480-frame) window after Sword to Answer Waves' Call before the Resonance Skill falls to cooldown.

A reserved **`swap-in`** token may appear in the array alongside real stage keys. It is **not** a resolvable stage key: compile passes it through untouched (real stageIds are dotted/colon-namespaced, e.g. `char.cartethyia.stage-1::...`, so the bare token can never collide), and the validator treats it as satisfied when the actor is a **fresh swap-in** — the immediately-preceding timeline entry is a different `characterId`, or this is the first entry. It models the near-universal tech where swapping a character in lets the first Normal Attack open at Basic Attack Stage 2 rather than Stage 1; that stage carries `requiresPriorStage: ["…/stage-1", "swap-in"]`.

The two checks live in different layers, matching the engine's existing division of labor. **Existence/ordering is validation-time** (`validate-timeline`, which stays frame-agnostic): chain mode does an immediately-preceding membership test (satisfied by any listed stage, or by `swap-in` on a fresh entry); window mode does a same-character backward scan for any listed prerequisite cast, hard-erroring if none exists. That guarantees the simulation will find an anchor. **Timing is simulation-time**, keyed off the prerequisite's recorded **cast frame** (stable across swap-cancel, unlike the last-hit frame): `followUpMinDelay` becomes a [[Padding Delay]] component `max`-combined with `swapBack` (both are absolute-frame floors on the same entry's start); `followUpMaxDelay` becomes an **invalid finding** when `cursor − anchorFrame > followUpMaxDelay`, symmetric to an unmet cooldown. Because `validate-timeline` deliberately does not compute absolute frames, the window-closed check cannot live there — it belongs to the sim, the one place that already tracks anchor cast frames. Multiple prior casts anchor to the **most recent** before the follow-up, so re-casting re-arms both clocks. In the common full-cast case (Flaming Woolies advances its full 108 ≥ the 103-frame `followUpMinDelay`) the min pad is **0** — the gate is free, biting only when the prerequisite was swap-cancelled and the actor returned early.

`followUpMinDelay` is **too soon ⇒ pad, never error**: like Swap-back Cooldown, it is a timing fact the player cannot violate, so the engine models the forced wait rather than red-flagging a legal intent. `followUpMaxDelay` is the opposite face — **too late ⇒ error**: the player _can_ miss the window, and missing it is a real authoring mistake, so it surfaces as a finding.

`comboAllows` remains retired (removed with the first generalization): it was speculative generality from ADR-0015 with zero authored consumers, and its removal was behavior-neutral. Window mode is a strictly more general expression of "intervening entries don't break the gate" than a movement whitelist ever was.

## Considered Options

- **Single-string `requiresPriorStage` (no array).** Superseded. It cannot express a follow-up reachable from several distinct predecessors: Cartethyia's Basic Attack Stage 2 is a legal follow-up to Stage 1, the Intro, the plunge, an Avatar form-swap, _and_ a fresh swap-in. OR-semantics over an array is the minimal expression of "any of these qualifies"; the string form survives as one-element sugar so no existing chain data churns.
- **A boolean `swapInOpener` flag instead of a `swap-in` array sentinel.** Rejected in favor of the sentinel. The flag keeps two concepts ("which prior stage" vs. "is this a swap-in opener") in two fields, which is tidier typing, but the sentinel gives a single, uniform authoring surface — the opener is just another alternative in the same OR-list the author is already writing — at the cost of one pass-through case in compile. The clear authoring interface won; the background handling is contained to a single reserved-token check.
- **Modeling Avatar form-swaps as literal `swap` timeline variants.** Rejected as too heavy. Cartethyia and Fleurdelys share one `characterId` (1409); making an Avatar a real swap would reach into on-field/trailing bookkeeping. Because the Avatar is an explicit timeline entry, listing it as an ordinary array member of the follow-up's gate reuses chain mode with no new machinery.
- **Explicit `mode: "chain" | "window"` discriminator instead of implicit delay presence.** Rejected. It would protect "orthogonal" axes — an immediate gate that also carries a delay — that are as speculative as `comboAllows`: WuWa's combo model is a max input buffer, not a min delay, making "immediate + delay" implausible. The one real subtlety — that a delay field also flips the gate non-positional — is captured in a type comment and `CHARACTERS.md`.
- **A separate field for the windowed case, leaving `requiresStageId` as-is.** Rejected. Two fields for one concept drift and force every consumer to check both. Generalizing the single field keeps one gate with one validator branch.
- **Keep `comboAllows`, only flip its default to always-transparent.** Rejected as half-measure. The field has no consumer; window mode subsumes the capability if it is ever needed.
- **Lose positional combo-breaking — make every gate windowed.** Rejected. `Stage 2 → Resonance Skill → Stage 3` is genuinely illegal in-game (a skill resets the basic chain) and chain mode models that; a purely temporal window would validate it green.
- **Compute delay timing in `validate-timeline` (hard error when too soon / too late).** Rejected. The validator deliberately does not compute absolute frames; teaching it to would duplicate the simulation's timing model in a second place that must be kept in sync. Both `followUpMinDelay` (pad) and `followUpMaxDelay` (window-closed finding) confine their frame math to the sim.
- **Anchor the delays on the prerequisite's last _hit_ frame rather than its cast frame.** Rejected. "After the cast" is the in-game phrasing, and the last-hit frame drifts under swap-cancel, which would make the anchor depend on variant and footing state. The cast frame is stable and recorded uniformly.
- **Continue to defer `followUpMaxDelay` (Clock B).** Superseded. The upper bound was deferred when no authored stage needed it and an unused field would invite speculative upkeep. Cartethyia's May Tempest is a concrete consumer with a known 8s window, so the field is added now, mirrored on the existing `followUpMinDelay` machinery rather than built as a bespoke mechanism.

## Consequences

- `EnrichedSkillAttribute`'s gated arm becomes `{ requiresPriorStage: string | string[]; followUpMinDelay?: number; followUpMaxDelay?: number }`; the ungated arm stays `{ requiresPriorStage?: never }`. Compile lowers `requiresPriorStage` to `requiresPriorStageId: string[]`, resolving each real key and passing the `swap-in` sentinel through.
- All prior `requiresStageId`/`followUpDelay` usages migrate field-name-only: Sanhua Frigid Light and Encore Wooly Attack / Cosmos: Frolicking keep bit-identical chain behavior; Encore's Energetic Welcome renames `followUpDelay: 103` to `followUpMinDelay: 103`.
- `validate-timeline` chain mode tests membership over the array and honors the `swap-in` sentinel via a fresh-entry check; window mode scans for any listed prerequisite. Messages carry the list of required stageIds.
- The simulation records prerequisite cast frames (most-recent-wins, per character + prerequisite stageId), computes the `followUpMinDelay` pad as a Padding Delay component, and emits an invalid finding when a `followUpMaxDelay` window has elapsed. Surfaced on the Timeline row and Simulation Log like the other pad/finding components.
- Every character's Basic Attack Stage 2 gains `swap-in` (adding the `stage-1` chain where a character previously had none), so the opener tech is modeled uniformly.
- `CHARACTERS.md` Cookbook updates the combo recipe to array form and adds a swap-in-opener recipe and a min/max-window recipe; `docs/enriched-model.md` updates the stage-field reference. The Stage-fields and "Rare fields" tables carry `requiresPriorStage: string | string[]`, `followUpMinDelay`, `followUpMaxDelay`, and the `swap-in` token.
- New/changed glossary terms: **Prior-Stage Gate** (array-valued, chain/window modes, `swap-in` sentinel); **`followUpMinDelay`** / **`followUpMaxDelay`** (the two window bounds); [[Padding Delay]] retains its `followUpMinDelay` component.
