# Swap variant as trailing-window with auto-pad and cancel-drop on same-character re-entry

The `swap` Stage Variant — deferred by [ADR-0008](0008-stage-variants-as-actionframe-truncation.md) pending the Trailing Window concept — ships as a third member of `VariantKind` alongside `cancel` and `instantCancel`. Unlike the other two, `swap` does **not** truncate the stage's damage: the cursor advances by a short `advance` value while every authored Damage Entry still resolves at its original `stageStartFrame + actionFrame`, including frames past the next entry's start. This is the **Trailing Window**: the swapped-out character's stage continues to emit hits in the background while the timeline cursor proceeds under whoever the player switched to (or the same character on a later re-entry).

To make the split expressible in the engine, `resolveStageExecution` returns `{ advance, hits }` instead of `{ duration, damage }`: for `cancel`/`instantCancel` the two numbers are equal and `hits` is filtered to `actionFrame ≤ advance` (the contract from ADR-0008 is preserved); for `swap` they decouple — `advance` is the cursor step and `hits` is the full authored list, unfiltered. The `advance` value for swap follows a **default-fallback** rule: if the stage authors `variants.swap = { actionTime: N }` the value is `max(N + reactionDelay, variantFloor)` (the same shape and floor as the other variants, ADR-0008); otherwise the value is the global `swapFrames` setting. `swapFrames` is a player-level field on `Settings` parallel to `reactionDelay`, defaulting to 6 frames at 60fps; both live in the existing `wuwa.settings` localStorage key. A `swap`-variant entry whose **immediately following** Timeline Entry has the same `characterId` raises a validation warning (not error) — swap exits the character from the field, so the next input cannot logically be the same character.

**Realization: trailing hits are stream members.** Trailing hits are not held in any per-character store; they are enqueued onto the one frame-ordered event stream (the `Schedule<T>` of [ADR-0028](0028-emithits-as-frame-honest-worklist.md)) with the **`residue`** arrival class and resolve frame-honestly at their `hitFrame`, interleaved with later entries' events. The immediate/trailing partition is the pure helper `partitionStage` in `src/lib/trailing-window.ts` (alongside `isCancelCapable`); per [ADR-0038](0038-post-actiontime-hits-resolve-in-frame-order-for-all-stages.md) the partition applies to **every** stage, with only swap windows enqueuing as cancellable `residue` (non-swap background hits use the uncollidable `ignore` class). A swap stage's launch/land footing commits ride the same stream — see [ADR-0022](0022-footing-as-team-state-with-trailing-window-snapshot.md).

**Same-character re-entry rule.** When a later entry shares `characterId` with a swap-variant stage whose trailing hits have not all landed, the engine branches on the re-entry's `skillType` via `isCancelCapable`:

- **Cancel-capable types** — `Resonance Skill`, `Resonance Liberation`, `Intro Skill`, `Outro Skill`, `Echo Skill`. The re-entry's own animation cancels the residual: no padding occurs, and the character's pending trailing hits at or past the re-entry's start are marked invalid (skipped at pop — no queue deletion; sound by causality, since a cancel only drops `hitFrame ≥ F` and so can never arrive after the hit it cancels). The drop is measured against the re-entry's **`effectiveStart`** — the cursor after `fall`/`swapBack`/`priorGate` pads — not the raw cursor, and the boundary hit landing exactly on that frame resolves first via the pre-drain; see [ADR-0037](0037-cancel-capable-residue-drop-at-effective-start.md). Cast-time resources on the original swap stage (cooldown, concerto, energy, resonance) were credited at its start, so no rollback. Dropped hits are silently discarded.
- **Non-cancel-capable types** — everything else (`Basic Attack`, `Heavy Attack`, `Tune Break`, `Movement`). The character is still locked in animation, so the re-entry's start is floored past the last trailing hit: `nextStart = max(cursor, latestPendingSameCharFrame)`, an arrival-time cursor rule on the authored walk. All trailing hits land. The added frames surface as pad attributed to the preceding entry.

The split is by `skillType` so that "normal attack combos cannot be cancel-interrupted" falls out of the type taxonomy rather than needing a stage-level `canInterrupt` flag. Hits dropped under the cancel-capable branch and frames added under the non-cancel-capable branch are mutually exclusive — exactly one applies per same-character re-entry. Different-character entries never trigger either path: their activity overlaps the trailing hits silently, by design.

**Padding/delay rendering.** Any frames added on top of a stage's base `advance` — whether from `reactionDelay`/`variantFloor` (every variant) or from a swap-back auto-pad — surface as a `+0.Xs` suffix on the Timeline row and the Simulation Log entry. The displayed number is the **merged** total; a tooltip splits the components (`react: 0.1s, pad: 0.3s`) for forensics. Display unit is seconds; the underlying values stay in frames.

**Trailing-hit ownership.** A trailing hit retains the swap stage's `characterId` as its acting character — the `TrailingHit` bundle captures the entry at scheduling time. Buff snapshotting reads whatever state is active at `hitFrame`: a team buff activated by the swapped-in character mid-window therefore _does_ affect trailing hits of the swapped-out character if the buff's filters match. The character remains **In-trailing** for the window's duration (the stage's `actionTime`); per-character footing during that state is ADR-0022's concern.

## Considered Options

- **Validation-only — `swap` is purely a UI label, engine treats it as no-variant.** Rejected. The user-visible behavior the variant exists to deliver — combo chains and damage continuing while other inputs happen on a short timeline cost — does not emerge without the cursor-advance / damage-cutoff split. A label that produces identical sim output would be cosmetic.
- **`swap` as truncation, identical mechanic to `cancel`.** Rejected for the reason ADR-0008 already gave: it silently deletes hits the player actually landed, producing systematically wrong DPS for swap-cancel rotations — the exact workflow the feature exists to enable.
- **A standalone Trailing Window state module (shipped twice, then dissolved).** The mechanic was first realized inline in `simulation.ts`, then extracted to a pure-function state machine over a per-character `TrailingWindowState` map (`onEntryArrival` / `scheduleStage` / `drainAll`), testable in isolation against this ADR's branching rules. Dissolved when ADR-0028's unification made every emission a member of one frame-ordered stream: a per-character side-store next to that stream duplicated ordering policy, resolved trailing hits in per-entry batches rather than true frame order (blocking the off-field-damage interleaving the stream exists for), and the drop/pad collision rules turned out to be expressible directly against the stream — drop as mark-invalid-at-pop, pad as a cursor rule. What survives of the module is `partitionStage` + `isCancelCapable`; the behavioral contract carried over verbatim.
- **A per-entry "N hits dropped" log annotation on the swap-canceled stage (shipped, then removed).** Removed because it required a `droppedHitCount` reach-back into an already-written Action Event and was temporally dishonest — drops happen at the new entry's frame, not at the swap stage. Cancel-capable trailing hits are silently discarded; forensics can return later as a dedicated log event emitted at drop time if a use case arises.
- **`swapFrames` as a literal floor (`advance = max(authoredActionTime + reactionDelay, swapFrames)`).** Rejected. `swapFrames` is a default for unauthored swaps, not a minimum for authored ones; the physical input-duration minimum is the separate `variantFloor` (ADR-0008), which applies uniformly to all authored variants.
- **`canInterrupt: boolean` flag on Stages to mark normal-attack chains as un-cancellable.** Rejected. Variants in ADR-0008 are already explicit per-stage opt-ins — a stage that should not be swap-cancellable simply omits `variants.swap`. The flag adds no information the absence of the variant doesn't already convey, and the auto-pad rule itself is type-driven (Basic Attack falls into the non-cancel-capable bucket), not stage-flag-driven.
- **Pad the swap stage itself instead of the re-entry's start.** Rejected. The swap stage's `advance` is the user-visible commitment of "I'm switching now"; retroactively inflating it to cover the trailing tail would obscure the gameplay reality and conflate the cursor-advance number with the animation-completion number, recreating the conflation this ADR exists to break.
- **Pad on all same-character re-entries regardless of `skillType`.** Rejected. The cancel-capable skill types genuinely interrupt residual animation in WuWa; padding them would over-charge frames the player doesn't actually spend, and the dropped-hits outcome is the in-game truth. The branch is load-bearing.
- **Drop trailing hits on _all_ same-character re-entries (no auto-pad anywhere).** Rejected for the symmetric reason: Basic-Attack continuations cannot interrupt a prior swap-cancel, so silently dropping hits would under-count damage the player actually lands.
- **Introduce on-field / off-field tracking now and key buff filters on it.** Rejected as scope creep at the time. No buff then authored differentiated on-field from off-field characters, and the only thing trailing hits needed was correct acting-character ownership. (The Deployment State model — On-field / In-trailing / In-reserve — later arrived with its first real consumer, per-character footing: ADR-0022.)
- **Validation _error_ (instead of warning) on same-character entry immediately after a swap.** Rejected. The warning is informational; a future authoring case may want the warning suppressed (e.g., demonstrating an invalid input to illustrate a constraint), and ADR-0008's pattern is that variants degrade gracefully rather than hard-blocking. Other immediate-prerequisite checks (`requiresPriorStageId`) are errors because the sim cannot proceed; here the sim can proceed and just produces a result the player flagged as physically impossible.
- **Per-entry `swapFrames` override.** Rejected for the same reason ADR-0008 removed the editable `actionTime` field: stage + variant + global settings derive the value, and a manual override creates a second source of truth that drifts.
- **Surface the pad/delay only in the Simulation Log, not on the Timeline row.** Rejected. The Timeline is the editing surface — the user needs to see at the row level why a stage is taking longer than its authored frames, otherwise the auto-pad behaves as "invisible engine magic." The merged `+0.Xs` suffix is compact enough not to clutter; the tooltip handles the split.
- **Surface dropped hits as a Timeline-row badge.** Rejected. Drops happen on the _prior_ (swap-canceled) entry, not the row the user is currently editing; the visual association is weak.

## Consequences

- `VariantKind` becomes `"cancel" | "instantCancel" | "swap"`. The Timeline variant toggle order in `TimelineEntryRow.tsx` becomes `Full → Cancel → Instant Cancel → Swap → Full`, with undefined variants skipped per existing logic. Label: `"SWAP"`.
- `Settings` carries `swapFrames` alongside `reactionDelay` (defaults 6 and 6). The localStorage key (`wuwa.settings`) is unchanged; older saved settings missing `swapFrames` fall back to the default via the `useLocalStorage` merge path.
- `resolveStageExecution` in `src/lib/stage/stage.ts` returns `{ advance, hits }`. The `cancel`/`instantCancel` numerics are preserved — `advance === oldDuration`, `hits === oldDamage` — so the truncation contract of ADR-0008 binds unchanged on those paths.
- `simulation.ts` partitions each stage via `partitionStage`, fires the immediate hits, and enqueues the trailing partition onto the `Schedule` (`residue` class for swap windows). The same-character collision policy lives in the `Schedule`'s arrival handling plus `cancelResidue` (ADR-0037); the pad stays a cursor computation on the authored walk.
- `validate-timeline` has a warning channel distinct from `rowErrors`. The same-character-after-swap rule emits into that channel only; `requiresPriorStageId` reachability treats any variant — including `swap` — as satisfying the gate, matching ADR-0008's rule.
- Cast-time effects (`skillCast` dispatch, cooldown roll, concerto, energy, resonance cost) fire on `swap` identically to `instantCancel`. The cast happened.
- Stages whose authored Damage Entries are all at `actionFrame: 0` derive no trailing window from `swap` — every hit lands at the cursor regardless. The variant remains authorable on such stages but produces the same DPS as `instantCancel` plus the `swapFrames` cursor cost. A backfill of realistic `actionFrame` data on swap-relevant stages is a prerequisite for swap to be meaningfully distinguishable on those stages, exactly as ADR-0008 already noted for `cancel`/`instantCancel`.
- Domain terms in the project glossary under Core simulation: **Swap (Variant Kind)**, **Swap Frames**, **Trailing Window**, **Padding Delay**.
