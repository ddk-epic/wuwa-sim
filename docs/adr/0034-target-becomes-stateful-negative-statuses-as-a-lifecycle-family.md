# The target becomes stateful and Negative Statuses are modelled as a ticking lifecycle family, not approximated attacker-side

Cartethyia is the first character whose damage depends on **state held on the enemy**: Aero Erosion is a stacking Negative Status that lives on the target, ticks Aero Erosion DMG on its own cadence, and is read by other effects (the inherent's vulnerability scales off its live stack count). Until now the engine had no enemy object at all — the target was static DEF/RES/level fed into the formula tail, and everything else was attacker-side. We introduce a real `Target` aggregate that holds the static enemy params **plus** a collection of live Negative Status instances, because the user judged the stack count load-bearing (the whole erosion archetype scales off it) and the primitive is reusable: S4 already names five sibling statuses (Spectro Frazzle, Electro Flare, Glacio Chafe, Fusion Burst, Havoc Bane), so a parameterized status family pays for itself immediately rather than speculatively.

A Negative Status is parameterized by a **`NegStatusDef`** (`{ type, element, cap, duration, tickInterval, baseUnit, stackFactor[] }`) and a live instance holds `stacks`, `cap`, `endTime`, and `nextTickFrame`. It exposes `apply(n)` / `reduceBy(n)` / `raiseToMax()` / `raiseCap(n)` and is **advanced by the engine frame cursor** the same way buff expiry already is — when the cursor passes `nextTickFrame` the instance emits one tick (reading the **live** stack count, since stacks change between ticks) and schedules the next. Aero Erosion's mechanic falls out of this directly: cap 3 (raisable), ~15s duration, ~2–3s interval, stacks that **do not decay individually** (the whole status expires at end of duration; re-applying at max only resets the timer). Empirically (a level-90 inflictor vs a level-100 enemy) per-tick damage is `baseUnit × stackFactor[stacks]` with `stackFactor = {1: 0.8, 2: 2, 3: 4, 4: 6, 5: 8, 6: 10}` (i.e. `2×(stacks−1)` for ≥2 stacks); `baseUnit` is the pre-mitigation value reverse-calculated through the test's `defMult × resistMult` so the live formula re-applies whatever enemy is configured.

The status family is its **own** base (reusing buff stacking/expiry as extracted utilities), **not** a shared `TimedInstance` hierarchy with `BuffInstance`. Buffs and statuses happen to both have an `endTime`, but their stacking semantics genuinely differ (refresh-at-max-only, mutable cap, no per-stack timers), and the load-bearing commonality is status-to-status, not status-to-buff. We rejected: (a) **approximating erosion attacker-side** (a self-buff assuming maintained max stacks, DoT as authored emit stages) — collapses the inherent's piecewise stack scaling and the cap manipulation into constants that drift and cannot be reused across the archetype; (b) **pre-expanding all ticks into the `Schedule` at apply time** (ADR-0028) — reuses existing machinery but fights its fire-and-forget arrival classes the moment duration resets or the interval changes, since there is no "cancel on status refresh" notion; (c) **a grand `TimedInstance` base** unifying buffs, statuses, and summons — over-reach: sword shadows are just presence-flag buffs on the existing path, and coupling the stable buff store to a new sibling buys symmetry the consumers do not want.

Modifiers that change a status's _behavior_ (Mandate of Divinity halves the Erosion tick interval) live **on the status instance**, set by a `negStatusMod` effect that writes `intervalMult` (and future status-property mods); the status multiplies its base interval by it when scheduling and reverts when the source buff expires. This deliberately keeps such modifiers **off the Stat Table** — the Stat Table is for damage-calc inputs only; a scheduling knob does not belong there, and "the modifier lives where the thing it modifies lives" needs no status→buff-engine query.

## Consequences

- **New `Target` aggregate** (a module within the existing single bounded context — _not_ a new context; the ubiquitous language does not diverge) holding static enemy params and the live Negative Status instances. `computeDamage` takes the target (see ADR-0035).
- **`NegStatusDef` + Negative Status instance** as a reusable, frame-cursor-advanced lifecycle family. Aero Erosion is the first member; the five statuses S4 enumerates drop in as data with no new engine code. The tick reads live stacks and produces a `StatusTick` damage source (ADR-0035).
- **`negStatus` write effect** (`apply N` / `reduceBy N` / `raiseToMax` / `raiseCap N`) joins the effect kinds, riding existing triggers (`skillCast`/`hitLanded`) so every stack-applying line in the kit (Intro +2, NA Stage 4 +1, Resonance Skill +2, S3, …) is data.
- **`negStatusMod` effect → `intervalMult` on the instance** homes Mandate's tick-interval cut without touching the Stat Table; extensible to other status-property mods.
- **`fromStatusStacks` value-expr** (sibling to ADR-0032's `scaledByStacks`) reads the target status's live stack count, so the inherent's vulnerability authors against it. The tick damage does _not_ use it — the status computes `baseUnit × stackFactor[stacks]` internally.
- **NegStatus-presence condition** — "targets with Negative Statuses" (the Outro) keys off the _presence_ of any status, independent of stacks or ticks.
- **`negStatusInflicted` trigger** joins the event taxonomy (ADR-0002), feeding S4's team buff (`target: global`) when any erosion-class status is inflicted.
- **Status family base is independent of `InstanceStore`** — buff stacking/expiry are reused as utilities; the working buff subsystem is untouched.

## Known limitations to revisit

- **`baseUnit` is reverse-calculated from a single empirical sample** (level-90 inflictor vs level-100 enemy) and needs the test character's Aero DMG Bonus / Deepen backed out to be clean; until then it is approximate, not authoritative.
- **Sword Shadows and form/Manifest are presence-flag buffs, not modelled state** — the form change is surfaced for log visibility only; renaming the acting character per Cartethyia⇄Fleurdelys switch is out of scope. Sword-shadow recall, Conviction, and the Heart/Mandate/Power-of-Discord virtues are abstracted as user-asserted toggles/conditions (Mandate = a self-buff carrying the Erosion Deepen + `intervalMult`), consistent with ADR-0032's "attack availability is the author's choice in a timeline-driven sim."
- **S1 is out of scope** — its Zeal/Crit-DMG ramp gates on _defeating_ targets, which the engine cannot represent.
- **Multi-target stack propagation** (Power of Discord, "raise to highest among targets") is inert in a single-target sim and unmodelled.
- Companion to ADR-0035 (the `vul` bucket and the polymorphic DamageSource the tick is one variant of).
