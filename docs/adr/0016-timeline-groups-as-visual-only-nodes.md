# Timeline Groups as visual-only nodes alongside entries

The Timeline gains a **Timeline Group** — a labeled header that owns a contiguous run of Timeline Entries for organizational purposes only. The persisted shape changes from `TimelineEntry[]` to `TimelineNode[]`, a discriminated union of `{kind:"entry", ...}` and `{kind:"group", id, label, locked, entries: TimelineEntry[]}`. The engine, `validateTimeline`, `TimelineSummary`, and row numbering all operate on the flattened list (`nodes.flatMap(n => n.kind === "group" ? n.entries : [n])`); groups never reach the simulator. Groups have an open/locked lifecycle: at most one group is open at a time, the open group captures every new sidebar-added entry as a member, and locking seals the boundary against drag-in / drag-out while still permitting internal reorder and per-entry deletion.

This ADR records the structural decision only. Visual/UX specifics (icon choice, menu vs inline actions, label styling, confirmation thresholds, lock-toggle chrome) are intentionally left out — they are expected to evolve with QA feedback without invalidating the decision recorded here.

## Considered Options

- **Shape B: flat `TimelineEntry[]` with an optional `groupId` field plus a sibling `TimelineGroup[]` array.** Rejected. The invariant the user cares about most — _the entries of a group are exactly these, in this order, and they stay contiguous_ — is free in the nested shape and must be policed in code in the flat shape. Empty groups are natural in the nested shape and awkward in the flat one (no entry to anchor the header render against). The cost of nesting is a one-time migration and an extra `flatMap` at engine seams; the cost of flat is two sources of truth for membership and ordering that drift over time.
- **Bracket markers in the array (`groupStart` / `groupEnd` nodes).** Rejected as a halfway position. With explicit lock-enforcement we _could_ keep brackets coherent through drags, but ungrouped entries between two groups become ambiguous (does the trailing entry belong to the prior group or stand alone?), and the drag invariants reduce to "membership is whatever the markers happen to bracket right now," which is exactly the policed-in-code problem we rejected Shape B to avoid.
- **Restricting a group to a single character's entries.** Rejected. The most common organizational unit in Wuwa rotations is the cross-character burst window — _Verina outro → Jinhsi swap-in → Jinhsi combo_ — and players name those as one phase. A per-character restriction would forbid the most useful label users want to write. Any per-character chrome can be derived from `group.entries` opportunistically without the structural constraint.
- **Group boundaries break combo chains for validation** (`requiresStageId` / `comboAllows`). Rejected. Groups are organizational, not semantic. Folding validation into the set of things groups affect would create spooky action at a distance — a timeline that runs identically in the engine would validate differently based on a purely cosmetic regrouping. Users who want phase separation that actually enforces have the engine-real option of inserting a swap or a Movement entry that breaks the chain by the existing rules.
- **Implicit ownership: "this group owns everything until the next group header or end of array."** Rejected. Combined with lock semantics it leaks: a user dragging an entry past a header would silently re-parent it into the next group with no visible boundary change. Explicit `entries: TimelineEntry[]` on the group node keeps ownership readable from the data alone.
- **Recursive groups (groups inside groups).** Rejected. For a damage-sim rotation — typically 10–40 entries — a single level of grouping covers the use cases, and flat keeps the type, render, and drag rules dramatically simpler. Extensible later if a deeper structure is genuinely needed.
- **Auto-lock the open group on save / page reload.** Rejected. Treating "open" as ephemeral session state silently destroys the user's in-progress intent on a tab refresh. Persisting `locked: false` as-is lets the user pick up exactly where they left off.
- **Multiple open groups (LIFO or most-recently-interacted).** Rejected. Creates the worst-case ambiguity for "where will my next sidebar click land?" — the very question the lock/open distinction was introduced to answer. At-most-one-open keeps the rule trivial.
- **One "Delete group" action that ungroups (keeps entries).** Rejected per user preference: the chosen semantic is "delete group + contents." The folder-deletion mental model is preferred over the bracket-removal one.

## Consequences

- `src/types/timeline.ts` introduces `TimelineNode` (discriminated union) and `TimelineGroup`. `TimelineEntry` keeps its current shape unchanged.
- `useTimeline` stores `TimelineNode[]` at the existing `wuwa.timeline.entries` localStorage key. `migrateEntries` (now `migrateNodes`) wraps any pre-existing `TimelineEntry[]` payload as `[{kind:"entry", ...e}, ...]`. New entries from the sidebar append into the open group's `entries` if one exists, otherwise to the top level.
- Every existing engine and validation seam that takes `TimelineEntry[]` continues to take `TimelineEntry[]` — they receive the flattened view. The flatten is a single shared helper (`flattenNodes(nodes)`) used by the simulator entry point, `validateTimeline`, `TimelineSummary`, and the simulation-log writer.
- Row numbering in `TimelineView` is computed against the flattened entry index, not the node index — group headers occupy a visual row but are not numbered. The display row at flattened-index `i` shows `i + 1`.
- `TimelineView` renders entry nodes as today and group nodes as a header row followed by the contained entry rows. The at-most-one-open invariant is enforced at the reducer: creating a new group while one is already open auto-locks the previous one.
- Drag semantics: dragging a group node moves its header and `entries` together to a top-level slot only (flat-only). Dragging an entry within its current group is allowed; dragging an entry across a group boundary (in or out) is rejected. Removing an entry via its existing ✕ shrinks the group naturally.
- `validateTimeline` is unchanged in logic and signature — it sees the flattened entry list. Combo gates (`requiresStageId`, `comboAllows` per ADR-0015) can cross group boundaries; groups are visual, not semantic.
- `SimulationLog` output is unaffected — the engine never sees groups, so the log carries no group references. Future export formats may choose to surface group structure as a presentation concern, but the log itself remains a flat sequence.
- Saved timelines from before this change load through the migration as a single flat list of entry nodes at the top level, with no groups. Users opt into grouping by clicking the "+ Group" affordance on existing rotations.
