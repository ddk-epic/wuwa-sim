# Page state is distributed through Jotai atoms, not hand-rolled context providers

`SimulatorPage` shares its state through a stack of bespoke `createContext` + `Provider` + `useXContext`-that-throws modules (`useTeamContext`, `useSettingsContext`, `useUiPreferencesContext`, `useRenamingGroup`), nested four deep around the page body. As provider coverage grew this became the dominant tech debt: ~40 lines of identical boilerplate per context, a wrapper pyramid that climbs with every new slice, and coarse value objects that re-render every consumer when any one field changes. We move distribution to Jotai atoms. A persisted object becomes one `atomWithStorage` base with derived field atoms layered on top; consumers subscribe to the derived atom they actually read, so a settings change no longer re-renders slot rows.

The reason Jotai over Zustand is the `TeamContext` polymorphism, not a generic preference. The same context is fed by two sources depending on tree position: the live `useTeam` in the sim, and a throwaway `useDraftTeam` in the Library's create modal, so the shared `TeamBuilderModal` body is written once and runs context-blind over both. Jotai's `<Provider>` models exactly this — "the same atoms, an isolated store instance" — so the draft survives as a bare `<Provider>` around the create modal while the boilerplate (`useDraftTeam`, `DraftTeamProvider`) is deleted. Zustand's clean form is a global singleton, which has no native answer for an isolated per-mount instance and would push the draft back into hand-rolled store-plus-context plumbing.

The collapse: the live team lives in a single explicit app store, provided once by a root `<Provider store>` in the shell rather than the implicit module-level default store (explicit lifecycle, and on the SSR shell one store is not silently shared across requests by accident); `settings` is a derived atom of the team (it stays per-team and rides Save/Load, see ADR-0040); UI preferences keep their own global `atomWithStorage` (the engine-input vs. UI-preference boundary of ADR-0027 is unchanged); the renaming-group id becomes a plain global atom (one timeline is ever mounted); `useSlot` becomes a `slotAtomFamily` so each slot row only re-renders on its own changes. `SimulatorPage` carries no providers of its own; the two `<Provider>`s app-wide are the root store provider and the draft-isolation provider scoping the create modal.

Persistence is left exactly as-is. The five localStorage keys map to distinct lifetimes and write granularities, and atoms (runtime distribution) do not change what must survive a reload. Merging the session keys would make every change rewrite the whole blob — the same coarse-object churn the derived atoms exist to avoid — so the keys stay split.

## Considered Options

- **Keep hand-rolled context, add `use-context-selector` (rejected).** Fixes the re-render axis but keeps the provider pyramid and all the boilerplate — it leaves the dominant pain (authoring/nesting) untouched.
- **Zustand global singleton stores (rejected).** Cleanest possible reads in a single-instance world, but the draft is not single-instance; recovering per-mount isolation means re-introducing a context wrapper around a hand-built store, the exact thing we are trying to delete.
- **Zustand, and drop the draft (rejected).** Make the create modal edit live state to keep everything a singleton. But the live team is `wuwa.team`-backed, so composing a new team would overwrite the loaded session on the first keystroke with no cancel — it deletes the isolation/cancel/atomic-commit guarantees, not just the boilerplate.
- **Jotai, draft as `<Provider>` scoping (chosen).** Deletes the boilerplate and the pyramid, keeps the draft's isolation for free, and gives derived-atom subscription granularity with no change to the storage format.

## Consequences

- The migration is two-pass and incremental. Pass one swaps distribution (context → atoms) leaf-first (renaming-group, then UI preferences, then settings, then team/`useSlot`), deleting each provider as its consumers move, so no step is a big-bang. Pass two folds the storage hooks into `atomWithStorage` per domain where the key format maps trivially.
- The `reviveActiveTeam` storage-boundary coercion stays the base atom's revive function, so the storage format and back-compat behaviour are unchanged through the migration.
- A known asymmetry is left untouched: `EditTeamModal` runs the builder over live state with no draft, so edits write through immediately with no cancel, unlike the create flow. Closing that gap is a separate design decision.
- `wuwa.simulation-log` remains persisted even though it is derivable by re-running the engine on load; dropping it is a product tradeoff (recompute flash, re-deriving the stored-signature staleness check), not something this change enables.
