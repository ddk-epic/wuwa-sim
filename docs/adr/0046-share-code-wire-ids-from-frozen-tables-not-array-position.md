# Share-code entity references are indices into frozen, append-only wire tables keyed by stable game ids, not positions in the live display arrays

The share codec used to encode every entity as its **index in a live array** — a character its position in `ALL_CHARACTERS`, a weapon in `ALL_WEAPONS`, a stage an ordinal into its character's compiled stage list. That made the format append-only _by luck_: inserting, removing, or reordering anything mid-array silently renumbered every code emitted after it, and the only guard was discipline plus a manual `VERSION` bump that invalidated all prior codes. The array that the UI reorders for display and the numbering the wire depends on were the **same list**, so any display edit was a wire edit.

The fix decouples the two. Each registry gains a **frozen, append-only `*_WIRE` table** — an ordered list of stable game ids (`CHARACTER_WIRE`, `WEAPON_WIRE`, `ECHO_WIRE`, `ECHO_SET_WIRE`, and a per-character `SKILL_WIRE`). The _index into that table_ is the wire byte; the display arrays (`ALL_CHARACTERS`, …) are then free to reorder, sort, or take mid-array inserts without touching the wire, because encode/decode look up by id through the table, never by array position. Adding an entity appends one row to its wire table and changes no existing index — a true append, no `VERSION` bump. The wire tables become the single frozen contract whose only job is byte stability; the display arrays become volatile again without consequence.

Stages are keyed on **skill id, not the derived stage string**. A stage's identity is `(skillId, stageOrdinalWithinSkill)`: every stage-bearing skill already carries a stable numeric game id, and a stage is its position inside that skill's compiled stages. This is deliberately chosen over freezing a snapshot of the `makeStageId(charSlug, categorySlug, skillKey, stageKey, skillType)` **strings**, because those strings are derived from names and keys — renaming a skill, renaming a stage, or recategorizing it would change the string and orphan every code referencing it. A skill id survives all of that. The residual coupling is narrow and accepted: reordering stages _within a single skill_ still shifts that skill's stage ordinals — a local, rare edit, inside the "100% decoupling is a non-goal" line.

A stage encodes as two bytes. The first is a **selector**: its high bit flags character-skill (`0`) versus echo (`1`), and its low seven bits are the index into that character's `SKILL_WIRE` (own) or the global `ECHO_WIRE` (echo). The flag rides a bit the index never needs (≤127 skills/echoes), so the own-vs-echo split costs nothing, and — unlike the old `own.length + echoIndex` scheme — appending a character skill never renumbers that character's echo-stage references. The second byte is the stage ordinal within the selected skill. That ordinal byte is **always written even when the skill has one stage**: eliding it for single-stage skills (most echoes) would make the wire _length_ depend on a skill's stage-count, a quantity that grows by append, so a skill later gaining a stage would misparse older codes — reintroducing exactly the append-fragility this change removes. Fixed width is the price of append-safety.

The `ECHO_BUILDS` / main-stat / variant enums (`BUILD_WIRE_ORDER`, `COST4_MAINS`, `COST3_MAINS`, `VARIANT_KINDS`) stay encoded as index-into-tuple. They are closed domains with no separate display array pulling them out of order, so the only risk is an accidental reorder in the codec file itself; they are brought under the same append-only guard test rather than given id-keyed wire tables they have no ids for.

The cutover is a **hard version cut**. `VERSION` becomes `6` and the decoder accepts only `6`; `3`/`4`/`5` codes are rejected with a re-export prompt rather than migrated. Preserving old codes was possible (seed the wire tables to the current order and they decode identically, even hardening against later reorders) but was declined — the roster is young, and a single clean decode path with no legacy `own.length + j` branch is worth more than compatibility with codes that predate the format.

Append-only is now **enforced, not merely documented**. A generator (`scripts/generate-wire-tables.ts`, emitting `wire-tables.generated.ts`) reads the committed frozen tables, keeps every existing row in place, and appends any live id not already present — deterministic and monotonic. Its `--check` mode regenerates and fails on any diff, catching a stale table (an unlisted live id) or a dropped live row. A guard test additionally asserts coverage (every live entity and skill resolves to a slot, so encode can never throw at runtime) and no duplicate ids per table. A removed entity keeps its row as a tombstone rather than being dropped, since dropping would shift later indices. A self-consistent reorder of the committed file can't be caught automatically — the generated file has no baseline but itself — so it is guarded by review of that committed artifact, not by the check.

## Considered Options

- **Encode the raw game id as a varint (fork-portable).** Rejected for now. A fork whose `ALL_CHARACTERS` differs from ours could still read the code, but game ids are 7–9 digits (3–4 bytes each) against the current single byte, and cross-fork portability is a further step nobody has asked for. The frozen tables leave a clean path to migrate any single registry to raw ids later if it ever matters.
- **Freeze a snapshot of the `makeStageId` strings as the stage wire table.** Rejected in favour of `(skillId, ordinal)`. It works, but the identifier is derived from names and keys, so every skill/stage/category rename orphans codes and piles up tombstones — and it costs ~60 KB of committed strings versus a tiny numeric `SKILL_WIRE`. Skill id is rename-proof and reuses the same numeric-table mechanism as every other registry.
- **Elide the stage-ordinal byte when the skill has a single stage.** Rejected. It saves a byte on most echoes but couples the wire length to a skill's stage-count, which grows by append; a skill gaining a second stage would then misparse every older code that omitted the byte — the precise fragility this change exists to remove.
- **Soft migration — seed the wire tables to the current order and keep decoding v3/4/5.** Viable and breaks nothing at cutover, but declined: the roster is young enough that invalidating existing codes is acceptable, and a single decode path with no legacy branch is simpler to keep correct.
- **Give every stage an explicit author-assigned id in the data.** Rejected as out of scope. It would decouple the within-skill ordinal too, but is a large data change for a coupling (reorder-within-one-skill) that is already local and rare.
- **Give the enum tuples id-keyed wire tables like the registries.** Rejected as overkill. They have no game ids, no separate display array reorders them, and the append-only guard test covers the only real risk.

## Consequences

- A new `src/lib/share/wire-tables.ts` holds the frozen `CHARACTER_WIRE`, `WEAPON_WIRE`, `ECHO_WIRE`, `ECHO_SET_WIRE`, and per-character `SKILL_WIRE`; `import-export.ts` routes every lookup through it instead of `findIndex`/array position.
- `VERSION` is `6`, decoder accepts only `6`; the `CHAR_STAGE_IDS` / `ECHO_STAGE_IDS` ordinal scheme and the `own.length + j` echo-stage offset are removed.
- A stage reference grows from one byte to two (selector + ordinal); a full timeline gains tens of bytes, unaffecting the slug prefix.
- `scripts/generate-wire-tables.ts` (with `--check`, emitting `wire-tables.generated.ts`) plus a guard test (coverage, no-dups, frozen enum prefixes) enter the test suite; appending a character, weapon, echo, skill, or stage now regenerates the tables rather than requiring a `VERSION` bump.
- The only order-dependence left in the format is reordering stages within a single skill; the append-only discipline the previous format documented but never enforced is now a failing test.
- Landed after the echo-builds consolidation, which introduces `BUILD_WIRE_ORDER` and moves the loadout slot-counts onto the `ECHO_BUILDS` record; that tuple joins the wire-tables module as another frozen citizen.
