# Concerto consumption via a cap-honoring `spend()` primitive and a `resourceConsumed` trigger

Concerto **accrual** stays uncapped (parallel to energy â€” see ADR-0003), reading past 100 so that wasted overcap is visible for optimization. The 100 bar is still a real cap â€” it is just honored at **spend**, not accrual. Every cost runs through one engine primitive, `spend(resource, cost)`: it silently clamps the value to the nominal cap (`CONCERTO_CAP = 100` for concerto, `maxEnergy` for energy), then subtracts `cost`, yielding `min(before, cap) âˆ’ cost` â€” what a real, capped player would hold. **Outro Skills** are the `cost = cap` case (drain to 0, unchanged); **character kits** spend partial amounts via data-authored `resource` `op: "sub"` effects that route through the same primitive. Buffs react to "any spend" through a dedicated `resourceConsumed` trigger rather than `resourceCrossed`, because crossing is threshold-keyed and cannot express "any net decrease, of any size" (no single threshold catches both a 130â†’0 Outro drain and a 100â†’70 partial spend); the trigger reports `amount = cost`, not the ledger's `before âˆ’ after`, so the overbank discarded by the clamp never inflates on-consume readers. We rejected (a) a hard concerto cap on accrual â€” it hides overcap, the exact signal we want to surface; (b) a literal `set(cap âˆ’ cost)` that ignores `before` â€” exact only at a full bar, it lets repeated below-cap spends never draw down (a 40-cost from 60 would stay at 60 forever); (c) `resourceConsumed` reporting the full `before âˆ’ after` decrease â€” inflates readers by the discarded overbank; (d) authoring the Outro drain per-character â€” the rule is universal, so it belongs in the engine next to the energy-on-Liberation precedent; and (e) reusing `resourceCrossed down` for readers â€” semantically lossy, silently missing partial spends.

## Consequences

- **Spends run through one `spend(resource, cost)` helper.** It clamps `before` down to the nominal cap (firing nothing for the discarded overbank), then subtracts `cost` through the ledger delta. The Outro path is a `skillCast` branch keyed on `skillCategory === "Outro Skill"`, sibling to the Resonance-Liberation energy block: it raises an `insufficientOutroConcerto` Diagnostic when `concerto < cost` (still firing â€” mirrors energy, ADR-0003's "cost gating not enforced" limitation; the Diagnostic rides `onEvent`'s result onto the Action Event and surfaces on the timeline row), then `spend(concerto, CONCERTO_CAP)` drains to 0 (surplus above 100 wasted by design). `CONCERTO_CAP = 100` is the single source of truth â€” both concerto's nominal cap and the Outro cost â€” replacing the old `OUTRO_CONCERTO_COST`.
- **The subtract fires crossings over the real bar; the clamp is silent.** The post-clamp subtract runs through the same pre-trigger-match dispatch ordering as the energy zero, so `resourceCrossed` down and `resourceConsumed` chain deterministically â€” but only across `min(before, cap) â†’ after`, never the fictional overbank above the cap, which the clamp discards without firing anything.
- **Partial spends are data, gated by author discipline.** Kits author `{ kind: "resource", resource: "concerto", op: "sub", value: { kind: "const", v: N } }` (already supported for `const` values) paired with `condition: { kind: "resourceAtLeast", resource: "concerto", n: N, on: "source" }`. The `op: "sub"` execution routes through `spend()`, so Outro (engine) and partial spends (data) now share the clamp-then-subtract primitive rather than only converging at the ledger. `op: "set"` is exempt â€” an authoritative write, not a cost.
- **Trigger `resourceConsumed`** joins the deliberately-grown event taxonomy (ADR-0002): a `Trigger` variant carrying `resource` and `amount`, a matcher in the trigger index, and a fire call alongside `fireResourceCrossed` gated on `after < before`. `amount` is the `cost` the spend drew, **not** the ledger's `before âˆ’ after`, so an overbanked spend (180 concerto, 70 cost â†’ 30) reports 70 consumed rather than 150, keeping on-spend readers (e.g. Red Spring's concerto-consumed passive) faithful to the real draw.
- **The ledger floors at 0.** `ResourceLedger.applyDelta` gains a low-side `Math.max(0, â€¦)` clamp alongside the existing high-side cap, so no resource can go negative regardless of authoring mistakes. General â€” also protects energy/forte.
