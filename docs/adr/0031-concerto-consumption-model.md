# Concerto consumption via engine-internal Outro drain and a `resourceConsumed` trigger

Concerto stays **uncapped** (parallel to energy тАФ see ADR-0003), accruing past 100 so that wasted overcap is visible for optimization. It is consumed two ways that share no code path but converge at the ledger: **Outro Skills** drain all concerto to 0 via engine-internal handling, and **character kits** spend partial amounts via data-authored `resource` `op: "sub"` effects. Buffs react to "any spend" through a new dedicated `resourceConsumed` trigger rather than `resourceCrossed`, because crossing is threshold-keyed and cannot express "any net decrease, of any size" (no single threshold catches both a 130тЖТ0 Outro drain and a 100тЖТ70 partial spend). We rejected (a) a hard concerto cap of 100 тАФ it hides overcap, the exact signal we want to surface; (b) authoring the Outro drain per-character тАФ the rule is universal, so it belongs in the engine next to the energy-on-Liberation precedent; and (c) reusing `resourceCrossed down` for readers тАФ semantically lossy, silently missing partial spends.

## Consequences

- **Outro drain is engine-internal.** The `skillCast` handler gains a branch keyed on `skillCategory === "Outro Skill"`, sibling to the Resonance-Liberation energy block: it raises an `insufficientConcerto` Diagnostic when `concerto < OUTRO_CONCERTO_COST` (still firing тАФ mirrors energy, ADR-0003's "cost gating not enforced" limitation; the Diagnostic rides `onEvent`'s result onto the Action Event and surfaces on the timeline row) and then `setResource(concerto, 0)`. Full drain, so surplus above 100 is wasted by design. `OUTRO_CONCERTO_COST = 100` is the single source of truth.
- **The drain fires crossings for free.** `setResource(тАж, 0)` runs through the same pre-trigger-match dispatch ordering as the energy zero, so `resourceCrossed` down and `resourceConsumed` chain deterministically.
- **Partial spends are data, gated by author discipline.** Kits author `{ kind: "resource", resource: "concerto", op: "sub", value: { kind: "const", v: N } }` (already supported for `const` values) paired with `condition: { kind: "resourceAtLeast", resource: "concerto", n: N, on: "source" }`. Outro (engine) and partial spends (data) stay distinct paths, unified only at `applyResourceDelta`/`setResource` so consumption events fire uniformly.
- **New trigger `resourceConsumed`** joins the deliberately-grown event taxonomy (ADR-0002): a `Trigger` variant carrying `resource` (and optionally amount), a matcher in the trigger index, and a fire call alongside `fireResourceCrossed` gated on `after < before`. This is the faithful hook for "on spend" effects (e.g. Red Spring's concerto-consumed passive).
- **The ledger floors at 0.** `ResourceLedger.applyDelta` gains a low-side `Math.max(0, тАж)` clamp alongside the existing high-side cap, so no resource can go negative regardless of authoring mistakes. General тАФ also protects energy/forte.

## Known limitations to revisit

- **Outro cost gating is warn-only, not a gate.** The static `validate-timeline.ts` is resource-agnostic (concerto is a sim-time quantity), so the "Outro needs тЙе 100" rule cannot be a structural row error. It surfaces instead as an `insufficientConcerto` Diagnostic on the entry's Action Event, shown on the timeline row under the stale regime тАФ the cast still proceeds and drains. Parallel to ADR-0003's Liberation-gating limitation.
- **Partial-spend `op: "sub"` handles only `value.kind === "const"`** (`applyResourceEffect` early-returns otherwise). Per-stack or scaled concerto costs are not yet expressible.
