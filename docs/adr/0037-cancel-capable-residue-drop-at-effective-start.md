# A cancel-capable re-entry cancels pending residue against its `effectiveStart`, not the pre-pad cursor

The same-character residue **drop** is moved off the **raw cursor** and onto the re-entry's **effective start frame** — the cursor after `fall`, `swapBack`, and `priorGate` are folded in. `resolveArrival` still runs at the input cursor, but only for the work that belongs there: cancelling a `reset` the owner caught early, and reporting the non-cancel-capable pad. The cancel-capable drop becomes a separate `Schedule.cancelResidue(owner, frame)` call, invoked at `effectiveStart` after the pre-drain to that frame. Previously a single `resolveArrival` did both jobs at `cursor.frame`, before any pad was known.

The motivating bug: `Flaming Woolies (swap) → Eternal Frost → Energetic Welcome`. Flaming Woolies' swap variant has `advance 0`, so all 8 hits (`actionFrame` 33…103) become trailing residue inside a 108-frame Trailing Window. Energetic Welcome is a `Resonance Skill` (cancel-capable) gated by `requiresPriorStageId: flaming-woolies` + `followUpDelay: 103`. After Eternal Frost (a 64-frame cast) the cursor sits at `S+64`; the old drop invalidated every residue hit with `frame ≥ S+64` — four woolies (`S+73…S+103`) — **before** the `followUpDelay` pad pushed Energetic Welcome's real start to `S+103`. Only 4 of 8 hits landed. In-game all 8 land: the player is still In-trailing for the whole pad, and the gate delays the _stage_, not the residual woolies. ADR-0036 assumed the In-trailing case needed no machinery because the window had "essentially closed" by `followUpDelay` — but the window is 108, the last hit and `followUpDelay` are both 103, and the collision was simply read at the wrong frame.

The fix separates two behaviors the old code fused, which turn out to have **opposite ordering needs**. A non-cancel-capable re-entry's pad _produces_ a floor — extend past the last trailing hit — and must be known **before** the start is finalized. A cancel-capable re-entry's drop _consumes_ the final start and must run **after** `fall`/`swapBack`/`priorGate` are applied. Resolving them at one frame forces one of the two to read a wrong frame; splitting them by frame is what makes both correct. This restores `CONTEXT.md`'s invariant — "trailing hits with `hitFrame ≥ newEntry.startFrame` are dropped" — to its intended meaning; it had silently meant `cursor.frame`.

The drop runs _after_ the pre-drain to `effectiveStart`, so a hit landing exactly on the start frame has already resolved and only the strict overlap is cancelled. This is the game truth for the repro: the 8th woolie lands as Energetic Welcome opens on the same frame.

`reset` and the residue drop are resolved at **different frames**, by necessity. The `reset`/footing resume must be read at the **input cursor**, because `fall` is computed from the footing that resume decides, and `fall` feeds `effectiveStart`; the drop is read at `effectiveStart`. `fall` itself stays **additive** — a transition cost paid once the actor is cleared to act, not a clock that overlaps the swap-back/gate wait. The narrow corner where `fall > 0` **and** a trailing hit lands inside the fall window is left undefined until an authored case needs it — the YAGNI posture ADR-0036 took toward `maxDelay`.

## Considered Options

- **Bump `followUpDelay` to 104 (data fix).** Rejected. The drop reads the pre-pad cursor (`S+64`), not the padded start, so moving the start a frame later changes nothing — the same four hits drop. It also misreads the failure as a one-frame tie when it is a wholesale drop of all residue at/after the early cursor.
- **Move the whole `resolveArrival` call after the pads.** Rejected. The `reset`/footing resume must run _before_ `fall` (it feeds footing), so the call cannot move wholesale. Splitting the input-cursor work (`reset` + pad) from the `effectiveStart` work (drop) is the only ordering that keeps both correct.
- **Re-express the entry start as an explicit `max` of floors, retiring the `rawPad − swapBack` term in `computePriorGatePad`.** Rejected _at the time_ as lateral churn unrelated to the drop-frame bug: the subtraction already realizes `max(swapBack, gate)`, and rewriting it while preserving the two displayed components changes no behavior. **Superseded** — see the Amendment in Consequences. The premise that the split is _displayed_ proved false (no consumer renders `swapBack` and `priorGate` apart), so the two floors collapse to a single `wait` and the encoding is retired after all.
- **Apply the `≥` drop before the pre-drain.** Rejected. That drops the hit landing exactly on `effectiveStart`, contradicting the 8-hits-land truth. Draining `≤ effectiveStart` first, then dropping, lands the boundary hit and cancels only the strict overlap.

## Consequences

- `resolveArrival` no longer drops: it cancels `reset` and returns the non-cancel-capable pad (both at the input cursor). A new `Schedule.cancelResidue(owner, frame)` performs the cancel-capable drop, called from `processEntry` at `effectiveStart` after the pre-drain. The `schedule.test.ts` residue cases split into `resolveArrival` (no drop) and `cancelResidue`.
- `CONTEXT.md` is corrected: the **Trailing Window** drop rule and **Schedule** residue class state the cancel is measured at `effectiveStart` with the boundary hit landing via the pre-drain; the **Prior-Stage Gate** entry drops the false "Trailing Window has closed anyway" justification.
- A regression test fixes `Flaming Woolies (swap) → Eternal Frost → Energetic Welcome`: all 8 woolie hits land (frames 33…103) and Energetic Welcome starts at `S+103`.
- Amends [ADR-0036](0036-prior-stage-gate-with-windowed-mode-and-mindelay-pad.md) (the In-trailing "no machinery" claim) and refines the cancel-capable arm of [ADR-0018](0018-swap-variant-as-trailing-window.md) / [ADR-0028](0028-emithits-as-frame-honest-worklist.md).

### Amendment — the two start-floors collapse to a single `wait`

A later audit of every read of `swapBack`/`priorGate` found they are **never surfaced apart**: `WaitBadge` sums them into one `wait 0.Xs`, and the timeline/log only OR-check `> 0` to decide whether to show the wait column. The "displayed split" the rejected `max` option above set out to preserve does not exist. So the two start-floors collapse to a single `DelayBreakdown.wait = max(swapBackWait, gateWait)`, computed once where the entry start is assembled. `computePriorGatePad`'s `swapBack` parameter and its `rawPad − swapBack` encoding are retired — it becomes `computeGateWait`, returning the raw gate wait, and the `max` lives at the assembly site. The change is behavior-identical: `effectiveStart = cursor + fall + wait` equals the previous `cursor + fall + swapBack + priorGate`, since `swapBack + max(0, gate − swapBack) ≡ max(swapBack, gate)`. The residue drop still measures at `effectiveStart`; this ADR's core decision is untouched. `fall` stays an additive `pad` component (the `+0.Xs` suffix), distinct from `wait` (the wait badge).
